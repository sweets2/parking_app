Plan: Systemic Street-Cleaning Geometry Fix via Per-Street Centroid Filter

=== CONTEXT ===

road-geometry.json was built from an OSM bbox larger than Hoboken. Numbered streets
contain both Hoboken ways and Jersey City ways. Two contamination clusters exist:

- South JC cluster: 9TH-13TH ST have ways at lat 40.728-40.731, INSIDE the Hoboken
  lat window (40.728-40.760). A rectangular mask cannot catch these.
- North JC cluster (JC Heights): 3RD-6TH ST have ways at lat 40.753-40.759, above
  the Hoboken expected lat for those streets.

Per-way avg-lat centroid data (seeds, Step 0 calibration is authoritative):

  2ND ST : Hoboken ~40.739 / north JC ~40.753
  3RD ST : Hoboken ~40.740 / north JC 40.753-40.755
  4TH ST : Hoboken ~40.741 / north JC 40.754-40.756
  5TH ST : Hoboken ~40.742 / north JC 40.756-40.758
  6TH ST : Hoboken ~40.744 / north JC ~40.757
  8TH ST : Hoboken ~40.747 / north JC ~40.758
  9TH ST : Hoboken ~40.748 / south JC 40.728-40.729
 10TH ST : Hoboken ~40.749 / south JC 40.728-40.729
 11TH ST : Hoboken ~40.750 / south JC 40.729-40.730
 12TH ST : Hoboken ~40.752 / south JC 40.730-40.731
 13TH ST : Hoboken ~40.753 / south JC 40.730-40.731, north JC 40.761
 14TH ST : Hoboken ~40.754 / southwest JC ~40.732
 15TH ST : Hoboken ~40.756 / southwest JC ~40.733
 16TH ST : Hoboken ~40.757 / southwest JC ~40.733

All centroid calculations use AVERAGE latitude (sum of pt.lat / count). Not min/max
midpoint. Consistent with existing snow-route filter.

The existing HOBOKEN_STREET_LAT / HOBOKEN_STREET_LAT_TOLERANCE = 0.008 pattern in
app/map.ts (snow routes, lines ~1665-1669) is the correct template. This plan
generalizes it to street-cleaning highlights.

Scope limitation: this fix covers numbered E-W streets only. Non-numbered streets
(GROVE ST, HENDERSON ST, NEWARK ST, etc.) may still have contamination. The audit
must enumerate remaining failures after this patch.

=== TWO BUGS ===

Bug 1: getBoundaryCoord uses raw unmasked geometry.
Inside parseLocationBounds, lats/lngs are computed from ALL ways including JC.
So "west boundary" returns the westernmost lon of every OSM way with that name.
Also latSpread/lngSpread (E-W vs N-S orientation detection) can be inflated.

Bug 2: renderViolationHighlights fallback is 1D (lat only).
When parseLocationBounds returns null, fallback clips by HOBOKEN_LAT_MIN/MAX (lat
axis only), leaving south JC ways at valid latitudes and west contamination.

--- Step 2: Add filterWaysByStreetLat helper ---

Place immediately before parseLocationBounds:

  function filterWaysByStreetLat(
    ways: [number, number][][],
    streetKey: string,
  ): [number, number][][] {
    const expected = HOBOKEN_STREET_LAT[streetKey];
    if (expected === undefined) return ways; // non-numbered: pass through
    return ways.filter(way => {
      if (way.length === 0) return false;  // guard: empty way has no centroid
      const centLat = way.reduce((sum, pt) => sum + pt[0], 0) / way.length;
      return Math.abs(centLat - expected) <= HOBOKEN_STREET_LAT_TOLERANCE;
    });
  }

Drops whole contaminated ways (no interpolation). Correct for cross-borough
contamination -- we don't want partial JC way segments.

--- Step 3: Apply inside parseLocationBounds ---

Before computing lats/lngs (currently lines ~1162-1170):

  // Before:
  const ways = _roadGeometry[streetKey];
  if (ways === undefined || ways.length === 0) return null;
  const allPts = ways.flat();

  // After:
  const rawWays = _roadGeometry[streetKey];
  if (rawWays === undefined || rawWays.length === 0) return null;
  const ways = filterWaysByStreetLat(rawWays, streetKey);
  if (ways.length === 0) return null;
  const allPts = ways.flat();
