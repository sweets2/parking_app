## F-20.5 Multi-Segment Highlight `[TODO]`

Amendment to F-20. Changes the segment-highlight system from single-match to multi-match so that all location blocks whose coordinate range brackets the click are highlighted simultaneously.

**Motivation:** On streets like Washington St., East and West sides are scraped as separate `StreetCleaningEntry` rows with different (overlapping) location ranges. A click between Observer Hwy and 7th St matches both the East-side block "Observer Hwy. to Seventh St." and the West-side block "Observer Hwy. to Eighth St." because `detectMatchingSegment` uses only the axis parallel to the street — it cannot determine which side of the street the user is on. Highlighting only the first match silently drops the other block. Highlighting all matches is the correct UX.

---

### Module: `app/main.ts` — update `buildDetectSegmentCallback`

Change the return type from `(locations: string[]) => Promise<string | null>` to `(locations: string[]) => Promise<string[] | null>`.

Replace the early-return loop with an accumulator:

```typescript
return async (locations: string[]) => {
  const matched: string[] = [];
  for (const location of locations) {
    const crossStreets = extractCrossStreets(location);
    if (crossStreets === null) continue;
    const [from, to] = crossStreets;
    const fromCoord = await geocodeCrossStreet(normalizeStreet(from));
    if (fromCoord === null) continue;
    const toCoord = await geocodeCrossStreet(normalizeStreet(to));
    if (toCoord === null) continue;
    if (detectMatchingSegment(clickLat, clickLng, fromCoord, toCoord)) {
      matched.push(location);  // collect all — do NOT return early
    }
  }
  return matched.length > 0 ? matched : null;
};
```

**Tests** — update existing tests in `tests/unit/main.test.ts`:
- The type cast for `detectSegment` changes from `Promise<string | null>` to `Promise<string[] | null>`.
- GIVEN `geocodeCrossStreet` returns coordinates that bracket the click, THEN `detectSegment` resolves to an **array** containing the matching location string (not a bare string).
- GIVEN `geocodeCrossStreet` returns `null` for all cross-streets, THEN `detectSegment` resolves to `null` (unchanged).
- GIVEN `extractCrossStreets` returns `null` for a location, THEN that location is skipped (unchanged).

### Module: `app/map.ts` — update `buildStreetPopupContent` and `showStreetPopup`

#### `buildStreetPopupContent`

Change signature from `highlightedLocation?: string` to `highlightedLocations?: string[]`.

Build an `activeSet` at function entry:
```typescript
const activeSet = new Set(highlightedLocations ?? []);
```

Change the per-block active check from:
```typescript
const isActive = highlightedLocation !== undefined && location === highlightedLocation;
```
to:
```typescript
const isActive = activeSet.size > 0 && activeSet.has(location);
```

Multiple blocks may carry `sp-block--active` at the same time.

#### `showStreetPopup`

Update the `detectSegment` parameter type:
```typescript
detectSegment?: (locations: string[]) => Promise<string[] | null>
```

The `.then` body is unchanged in structure — pass `matched` (now a `string[]`) directly to `buildStreetPopupContent`.

**Tests** — update existing and add new tests in `tests/unit/map.test.ts`:

- The stale-token test resolves with `["9th St. to 10th St."]` instead of a bare string; its type annotation changes from `Promise<string | null>` to `Promise<string[] | null>`.
- GIVEN `detectSegment` resolves to `null`, THEN `popup.setContent` is NOT called (unchanged).
- NEW — GIVEN `detectSegment` resolves to two location strings, THEN the popup content contains `"sp-block--active"` exactly twice.
