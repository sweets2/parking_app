## F-15 Manual Refresh Button `[TODO]`

A "Refresh signs" button visible only in parked mode. Shows data freshness. Tapping re-fetches and re-evaluates the saved spot.

This feature modifies `app/ui.ts` (button rendering and freshness label) and `app/main.ts` (fetch-and-update wiring). The fetch logic lives in `main.ts` — `app/ui.ts` is rendering only, and `app/app.ts` is not modified (CLAUDE.md: state machine only, no side effects).

### Module interface — new/modified exported symbols

```typescript
// app/ui.ts additions:

/** Renders the refresh button with a freshness label. Pass fetchedAt as an ISO string. */
export function renderRefreshButton(fetchedAt: string, now: Date): void

/** Sets the refresh button to a disabled/loading state. */
export function setRefreshLoading(loading: boolean): void

/** Shows a brief error toast: "Could not refresh — using cached data." */
export function showRefreshError(): void
```

The `app/main.ts` wires a click listener on `#refresh-btn` that calls `fetch("data/latest.json", { cache: "no-cache" })`, updates the app's sign data, calls `app.tick(new Date())`, and calls `setRefreshLoading` before/after. Existing behavior of `initBrowserApp` and all other functions is preserved unchanged.

---

### F-15.1 Button Visibility and Freshness Label

`renderRefreshButton` inserts (or updates) a `<button id="refresh-btn">` into the DOM. In parked mode it is visible (`display` not `"none"`). In browsing mode it is hidden (`display: "none"`) or not in the DOM. The button's text content or a child element includes the freshness label.

**Tests**
- GIVEN `renderRefreshButton` is called with any ISO string and a `now` date, THEN `document.getElementById("refresh-btn")` is non-null.
- GIVEN `renderRefreshButton` is called, THEN the button element's `style.display` is not `"none"` (it is visible).
- GIVEN `renderBrowsingMode` is called (browsing state), THEN `document.getElementById("refresh-btn")` is either null or has `style.display === "none"`.
- GIVEN `fetchedAt` is today at 06:02 UTC and `now` is the same day, WHEN `renderRefreshButton` is called, THEN `document.getElementById("refresh-btn").textContent` contains `"today"` and `"6:02"`.
- GIVEN `fetchedAt` is the previous day (e.g., `new Date(NOW_STABLE.getTime() - 24 * 60 * 60 * 1000).toISOString()`) and `now = NOW_STABLE`, WHEN `renderRefreshButton` is called, THEN the button text contains the 3-letter weekday abbreviation for that day (e.g., `"Mon"`, `"Tue"`, `"Wed"`, `"Thu"`, `"Fri"`, `"Sat"`, or `"Sun"`) and does NOT contain `"today"`.

---

### F-15.2 Tap to Refresh

Tapping `#refresh-btn` triggers a no-cache fetch in `main.ts`. The fetch pipeline (`filterLoadTimeNoise → filterActive → filterNearby`) runs on the fresh data, then `app.tick` re-renders state.

**Tests**
- GIVEN the refresh button click handler fires and the fetch resolves, THEN `fetch` was called with `"data/latest.json"` and `{ cache: "no-cache" }` (verified via `vi.spyOn(globalThis, "fetch")`).
- GIVEN the fetch resolves with a response containing 3 signs, WHEN the handler completes, THEN `renderSignPins` or `renderWarningBanner` is called with data reflecting those 3 signs (mock-based: `vi.mock("../../app/map")` and `vi.mock("../../app/ui")`).
- GIVEN `setRefreshLoading(true)` is called, THEN `document.getElementById("refresh-btn")` has `disabled === true`.
- GIVEN `setRefreshLoading(false)` is called, THEN `document.getElementById("refresh-btn")` has `disabled === false`.
- GIVEN the fetch fails with a network error, WHEN the handler catches the error, THEN `showRefreshError` is called and `setRefreshLoading(false)` is called after the error.
- GIVEN `showRefreshError` is called, THEN a DOM element containing the exact text `"Could not refresh — using cached data."` is visible in the document body.
