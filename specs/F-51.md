# F-51 ŌĆö Check Result Renderer

## Purpose

Render Check mode segment results on the map with block/side-scoped geometry and allow selecting
a segment.

## Mode affected

Check.

## State fields read

- activeMode
- checkResults
- selectedCheckSegment
- parkingSegments

## State fields written

- checkResults
- selectedCheckSegment

## TypeScript contract signatures

In `app/map.ts`:

```typescript
export function clearCheckResults(): void
export function renderCheckResults(results: CheckResultSegment[]): void
export function selectCheckSegment(id: string): void
```

### renderCheckResults

`renderCheckResults` draws block/side-scoped Leaflet polylines for each evaluated segment.
Geometry is resolved from `_roadGeometry` (the module-level road geometry loaded via
`initRoadGeometry`) ŌĆö **not** from `result.geometry` on `CheckResultSegment`. `result.geometry`
is a full-street lookup (`clipped: false`) and must not be used for Check rendering.

Geometry resolution per segment:
1. `normalizeToGeometryKey(result.street)` ŌåÆ look up in `_roadGeometry`
2. `parseLocationBounds(result.location, streetKey)` ŌĆö if null (location is not "X to Y" format),
   skip the segment silently (no polyline)
3. `clipWaysToBounds(allWays, bounds.min, bounds.max, bounds.axis)` ŌĆö if empty after clip, skip
4. Apply curb-side offset via `buildWayPolylines` (same zoom-adaptive logic as `drawWaysHighlight`)

Color/style map:

| status    | color    | opacity | geometry source      |
|-----------|----------|---------|----------------------|
| `"safe"`  | `#22c55e`| 0.40    | resolveCheckCurbWays |
| `"ticket"`| `#ef4444`| 0.60    | resolveCheckCurbWays |
| `"tow"`   | `#dc2626`| 0.75    | resolveCheckCurbWays |
| `"unknown"`| `#94a3b8`| 0.40   | resolveCheckCurbWays |
| `"snow"`  | ŌĆö        | ŌĆö       | skipped (not produced by evaluateParkingWindow) |
| `"limited"`| ŌĆö       | ŌĆö       | skipped (not currently produced) |

**Tow conflicts whose `location` field is a sign address (not parseable as "X to Y") produce no
Check stripe** ŌĆö `resolveCheckCurbWays` returns null and the segment is silently skipped. The
existing `renderTowSegments` visual marks those signs on the map. This is intentional; do not
re-add sign-address subsegment logic to `renderCheckResults`.

**Snow and limited status produce no Check stripe.** Snow is not produced by
`evaluateParkingWindow` (no real-time emergency data). Limited is not currently produced.

`renderCheckResults` calls `clearCheckResults()` at the top before drawing.

Side offset: use `result.side` when it is `"North"`, `"South"`, `"East"`, or `"West"`. Use
`null` (centerline) when `result.side` is any other value.

A module-private `Map<string, LeafletPolyline[]>` named `_checkLayers` stores id ŌåÆ polylines.

### buildWayPolylines (private helper)

A side-effect-free function shared between `drawWaysHighlight` and `renderCheckResults`:

```typescript
function buildWayPolylines(
  ways: [number, number][][],
  color: string,
  opacity: number,
  side: string | null,
): LeafletPolyline[]
```

- Calls `mergeWays(ways)`
- Computes zoom-adaptive lane width (`mPerPx`, `tileBoost`, `effectiveLaneM`, `laneWidthPx`,
  `weight`, `offsetM`)
- Applies `offsetPolylinePoints` when `side !== null`
- Calls `L.polyline(pts, { color, weight, opacity })`
- Returns `LeafletPolyline[]`
- Has **zero side effects**: does not touch `_violationLayers`, `_checkLayers`, `_map`, or any
  module state

### clearCheckResults

Removes all polylines from the map and empties `_checkLayers`.

### selectCheckSegment

Calls `setStyle({ className: "check-selected" })` on every polyline stored under that id in
`_checkLayers`. All other polylines are unaffected.

## Check legend

A `<div id="check-legend">` panel in `#legend-stack` (index.html) shows three swatches:
- Green solid ŌĆö Safe
- Red solid ŌĆö Street cleaning / Ticket
- Dark red solid ŌĆö Tow zone

Visibility is toggled in `renderState()` in `main.ts` alongside the existing `checkControls` /
`rulesControls` blocks: `display = activeMode === "check" ? "" : "none"`.

## state/event/rendering behavior

When the user runs a Check query:
- evaluate parking window for each parking segment
- store checkResults
- call `renderCheckResults(checkResults)`

Selecting a segment:
- updates selectedCheckSegment
- calls `selectCheckSegment(id)`
- later bottom sheet features use this state

## Acceptance criteria

Given checkResults contains a safe segment with id "seg-safe" and a ticket segment with
id "seg-ticket", and road geometry is available with parseable location bounds for both
When renderCheckResults([safeSegment, ticketSegment]) runs
Then a green polyline (#22c55e) is drawn for "seg-safe" and a red polyline (#ef4444) for
"seg-ticket"; `_checkLayers` has size 2.

Given a segment whose street is not in _roadGeometry (or whose location cannot be parsed as
"X to Y")
When renderCheckResults([segment]) runs
Then no polyline is drawn and _checkLayers remains empty.

Given a "tow" segment with road geometry and parseable location
When renderCheckResults([towSegment]) runs
Then 1 polyline is drawn with color "#dc2626" using block-scoped resolveCheckCurbWays geometry.

Given a segment with id "seg-ticket" has been rendered
When selectCheckSegment("seg-ticket") runs
Then setStyle is called with { className: "check-selected" } on that polyline.

Given activeMode switches to rules
When renderState runs
Then clearCheckResults() has been called and check-legend is hidden.

## Preservation requirements

- Preserve Rules layer stubs.
- Preserve map initialization.
- Preserve Check | Rules navigation.
- Do not use result.geometry for rendering ŌĆö it is full-street and reproduces the whole-street
  painting bug.
