## F-21 Build-Time Cross-Street Geocache `[TODO]`

`buildDetectSegmentCallback` (from F-20) geocodes cross-street names at runtime via Nominatim. Because Nominatim enforces 1 req/s and requests are sequential, the first click after opening the app can take several seconds to highlight the matching segment. This feature moves geocoding to build time: a one-shot script produces `data/cross-streets.json`, the app loads it at startup to seed the in-memory cache, and runtime Nominatim calls for cross-streets become unnecessary for the common case.

Three files are added or modified:

- `fetcher/geocode-cross-streets.ts` — build-time Node script (new file)
- `app/geo.ts` — add `seedGeocodeCache` export
- `app/main.ts` — fetch `data/cross-streets.json` at startup, call `seedGeocodeCache`
- `package.json` — add `geocode-cross-streets` script; extend `build` to copy `cross-streets.json`

`buildDetectSegmentCallback` and `geocodeCrossStreet` are unchanged. The cache check in `geocodeCrossStreet` already returns instantly on a hit, so pre-seeded entries are served with zero network calls.

---

### Module: `fetcher/geocode-cross-streets.ts` (new file)

A one-shot Node/tsx script that mirrors the pattern of `fetcher/fetch-street-cleaning.ts`.

**Exported symbols**

```typescript
/** Main entry point — reads data/street-cleaning.json and writes data/cross-streets.json. */
export async function runGeocode(): Promise<void>
```

**Algorithm**

1. Read `data/street-cleaning.json` (relative to the project root via `path.resolve`).
2. For every entry call `extractCrossStreets(entry.location)`. If it returns `null`, skip the entry.
3. Call `normalizeStreet` on each half of the pair. (Duplicate the `normalizeStreet` function from `app/main.ts` — importing `app/main.ts` is off-limits because it imports Leaflet, which requires a browser DOM.)
4. Collect all unique normalized cross-street names into a `Set<string>`.
5. Geocode each name against Nominatim at 1 req/s using a module-level rate-limit variable. Use the same Nominatim endpoint as `geocodeCrossStreet` in `geo.ts`: `https://nominatim.openstreetmap.org/search?format=json&q=<name>, Hoboken, NJ&limit=1`.
6. **Critical:** Set `User-Agent: "hoboken-parking-app/1.0 (build-time geocoder)"` on every request. Node's built-in `fetch` sends no User-Agent and Nominatim returns HTTP 403 without one. The browser sends a User-Agent automatically, which is why `geo.ts` works at runtime without it.
7. Write `data/cross-streets.json` as a `Record<string, { lat: number; lng: number } | null>`. Keys are normalized street names; values are `{ lat, lng }` or `null` for streets Nominatim could not resolve.

**Run guard** — mirror the pattern in `fetch-street-cleaning.ts`: run `runGeocode()` only when `process.argv[1]` ends with `geocode-cross-streets.ts` or `geocode-cross-streets.js`.

**Output format**

```json
{
  "willow avenue": { "lat": 40.7371, "lng": -74.0348 },
  "north boundary": null
}
```

**Tests** — none required for this file. It is a side-effectful I/O script; correctness is verified by running it and inspecting the output.

### Module: `app/geo.ts` — add `seedGeocodeCache` export

```typescript
export function seedGeocodeCache(
  table: Record<string, { lat: number; lng: number } | null>
): void
```

Iterates `Object.entries(table)`. For each `[key, value]`, if `_crossStreetCache.has(key)` is false, call `_crossStreetCache.set(key, value)`. Do NOT overwrite existing entries — a live geocode result takes priority over a stale build-time value.

**Why the guard is correct:** `geocodeCrossStreet` stores `null` on failure (`_crossStreetCache.set(streetName, null)`), so the cache uses `has()` not truthiness for its lookup. Seeding `null` for an unknown street correctly prevents a future Nominatim call for it.

**Tests** (append to `tests/unit/geo.test.ts`)

- GIVEN `seedGeocodeCache` is called with `{ "willow avenue": { lat: 40.737, lng: -74.034 } }`, WHEN `geocodeCrossStreet("willow avenue")` is called, THEN it returns `{ lat: 40.737, lng: -74.034 }` without making a fetch call.
- GIVEN `seedGeocodeCache` is called with `{ "north boundary": null }`, WHEN `geocodeCrossStreet("north boundary")` is called, THEN it returns `null` without making a fetch call.
- GIVEN `geocodeCrossStreet("willow avenue")` has already been called and returned a live result `{ lat: 1, lng: 2 }`, WHEN `seedGeocodeCache` is called with `{ "willow avenue": { lat: 9, lng: 9 } }`, THEN `geocodeCrossStreet("willow avenue")` still returns `{ lat: 1, lng: 2 }` (live result is not overwritten).

### Module: `app/main.ts` — startup cache seed

Import `seedGeocodeCache` from `./geo`.

In `initBrowserApp()`, alongside the existing fire-and-forget fetch of `data/street-cleaning.json`, add a second parallel fire-and-forget fetch:

```typescript
fetch("data/cross-streets.json")
  .then((res) => res.json())
  .then((data: unknown) => {
    seedGeocodeCache(data as Record<string, { lat: number; lng: number } | null>);
  })
  .catch(() => { /* non-fatal — runtime Nominatim calls serve as fallback */ });
```

The fetch must be issued before or at the same time as the `latest.json` fetch so the cache is seeded before the user's first click. Do NOT add this fetch to the `init()` function — that path is used by tests and does not need the browser data fetch.

**Tests** — no new tests required. The existing `main.test.ts` coverage verifies `buildDetectSegmentCallback` behavior; cache seeding is covered by `geo.test.ts`.

### Module: `package.json` — script additions

Add to `scripts`:

```json
"geocode-cross-streets": "tsx fetcher/geocode-cross-streets.ts"
```

Update the `build` script to also copy `data/cross-streets.json` into `app/data/`. The copy must be non-fatal (the file may not exist on a fresh checkout before the script is run):

```
&& (cp data/cross-streets.json app/data/cross-streets.json 2>/dev/null || true)
```

**Run order** — `geocode-cross-streets` must be run after `fetch-cleaning` because it reads `data/street-cleaning.json`. It does not need to run on every build — only when the cleaning schedule changes.
