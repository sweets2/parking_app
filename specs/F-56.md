# F-56 — Check Bottom Sheet + Desktop Responsive Polish

## Purpose

Complete Check mode detail presentation and polish responsive layout for both mobile and desktop.

## Mode affected

Check and Rules.

## State fields read

- activeMode
- selectedCheckSegment
- checkResults
- rulesInspectionSections

## State fields written

None directly.

## TypeScript contract signatures

```typescript
// app/ui.ts — mandatory export
export function renderCheckSegmentDetails(
  segment: CheckResultSegment
): string;
```

`renderCheckSegmentDetails` returns an HTML string containing the segment's street, side, location, status class, and primary reason. It must be importable from `app/ui.ts` and covered by unit tests.

## DOM IDs/classes or visible copy

Check bottom sheet content must include:
- street name text
- side text (e.g. "East", "West")
- location/range text (e.g. "Observer Hwy to 4th St")
- a status element with class `status-<status>` (e.g. `status-ticket`, `status-tow`, `status-safe`)
- primary reason text from `primaryConflict.reason` when present
- conflict time window text from `primaryConflict.label` when `primaryConflict` is defined
- next restriction label from `NextRestriction.label` when passed

Responsive classes present in CSS (must exist in `app/style.css`):
- `.layout`
- `.layout--desktop`
- `.bottom-sheet`
- `.bottom-sheet--check`
- `.bottom-sheet--rules`
- `.status-safe`
- `.status-unknown`
- `.status-limited`
- `.status-ticket`
- `.status-tow`
- `.status-snow`

## state/event/rendering behavior

When a Check segment is selected:
- `selectedCheckSegment` is read
- bottom sheet mode becomes Check
- bottom sheet opens
- `renderCheckSegmentDetails` is called and its output is placed in `bottom-sheet-content`

Desktop:
- map remains visible
- controls do not sprawl
- bottom sheet/panel remains readable

Mobile:
- bottom sheet does not cover the whole map by default
- controls remain touch-friendly
- Check | Rules nav remains accessible

## Acceptance criteria

Given a `CheckResultSegment` with `status: "ticket"` and `primaryConflict: { status: "ticket", reason: "Street Cleaning", label: "Mon–Fri 8–9 AM", startsAt: undefined, endsAt: undefined }`
When `renderCheckSegmentDetails(segment)` is called
Then the returned string contains the substring `"status-ticket"` and the substring `"Street Cleaning"`.

Given a `CheckResultSegment` with `side: "East"` and `street: "Observer Hwy"` and `location: "1st St to 2nd St"`
When `renderCheckSegmentDetails(segment)` is called
Then the returned string contains the substring `"East"`.

Given a `CheckResultSegment` with `location: "Observer Hwy to 4th St"`
When `renderCheckSegmentDetails(segment)` is called
Then the returned string contains the substring `"Observer Hwy to 4th St"`.

Given a `CheckResultSegment` with `status: "safe"` and no `primaryConflict`
When `renderCheckSegmentDetails(segment)` is called
Then the returned string contains the substring `"status-safe"` and does not contain `"status-ticket"`.

Given a `CheckResultSegment` with `primaryConflict: { status: "ticket", reason: "No Parking", label: "Tue 7–8 AM", startsAt: undefined, endsAt: undefined }`
When `renderCheckSegmentDetails(segment)` is called
Then the returned string contains the substring `"Tue 7–8 AM"`.
