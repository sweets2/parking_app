# F-49 — Segment Catalog + Geometry Utils

## Purpose

Build explicit parking segments from signs, street-cleaning entries, snow routes, and road geometry. The catalog must preserve street side and block/location specificity.

## Mode affected

Check and Rules.

## State fields read

None.

## State fields written

None directly. Later features store parkingSegments in ready state.

## TypeScript contract signatures

Create shared/segment-catalog.ts.

Required exports:

```typescript
export function buildParkingSegmentCatalog(input: {
  signs: Sign[];
  cleaningEntries: StreetCleaningEntry[];
  snowRoutes: SnowRoute[];
  roadGeometry?: RoadGeometry;
}): ParkingSegment[];

export function makeParkingSegmentId(input: {
  street: string;
  side: ParkingSide;
  location: string;
}): string;

export function normalizeSegmentToken(value: string): string;
```

## Behavior

Segment identity must include:
- street
- side
- location/range

Do not collapse all rules for one street into one segment.

Washington St hard rule:
If Washington St has different rules by side or block, return separate segments.

Unknown side:
- Use side: "Unknown".
- Do not guess side.

Unknown location:
- Preserve available source text.
- Do not merge into a citywide generic segment unless no better data exists.

Geometry:
- Attach SegmentGeometry when road geometry can be matched.
- If geometry cannot be matched, still return the logical segment (geometry field is absent/undefined).

## Matching algorithm

Each source (cleaningEntries, towSigns, snowRoutes) contributes segments independently:

- **cleaningEntries**: Each `StreetCleaningEntry` with a distinct `(street, side, location)` triple produces one `ParkingSegment`. Match is by street name and side string; location is taken directly from `entry.location`. The segment's `cleaningEntries` array contains all entries sharing that triple.
- **towSigns**: Each `Sign` with a distinct `(street, side, location)` triple produces one `ParkingSegment`. Match is by sign's `street` and `side` fields. The segment's `towSigns` array contains all signs sharing that triple.
- **snowRoutes**: Each `SnowRoute` with a distinct `(street, side)` pair produces one `ParkingSegment`. The segment's `snowRoutes` array contains all routes sharing that pair. Location is constructed as `"${route.from} to ${route.to}"`.
- If two source types produce a segment with the same `(street, side, location)` triple (after normalization), they are merged into one `ParkingSegment` and their arrays are combined.

## Acceptance criteria

Given two Washington St entries on different sides
When buildParkingSegmentCatalog runs
Then it returns two different segment IDs.

Given Washington St has entries for "Observer Hwy to 4th St" on one side and "4th St to 8th St" on another side
When buildParkingSegmentCatalog runs
Then it returns at least 2 segments with distinct IDs (one per unique street/side/location combination).

Given side is "Unknown"
When makeParkingSegmentId({ street: "Washington St", side: "Unknown", location: "Observer Hwy to 4th St" }) is called
Then the returned string contains the substring "unknown".

Given location is "Observer Hwy to 4th St"
When normalizeSegmentToken("Observer Hwy to 4th St") is called
Then it returns "observer-hwy-to-4th-st".

Given road geometry is unavailable (roadGeometry is undefined) and cleaningEntries has 1 entry
When buildParkingSegmentCatalog runs
Then it returns exactly 1 ParkingSegment with geometry undefined and cleaningEntries.length equal to 1.
