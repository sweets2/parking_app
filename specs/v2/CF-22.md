# CF-22 — Runtime Rendering from Precomputed Cleaning Segments

## Module
`app/map.ts`, `app/data-loader.ts`, `app/main.ts`, `tests/unit/map.test.ts`,
`package.json`

## Problem
`renderViolationHighlights` currently infers segment geometry live from raw
`StreetCleaningEntry[]` data using `parseLocationBounds`/`getCrossStreetCoord`/
`clipWaysToBounds`. CF-21 produced `data/cleaning-segments.json` with precomputed arc
positions. This feature wires the precomputed data into the runtime renderer.

## Goal
Load `cleaning-segments.json` at startup. Update `renderViolationHighlights` to render
from precomputed `CleaningSegment[]` instead of inferring geometry from raw entries.
Keep the old inference code path temporarily behind a feature flag for comparison.
Preserve all address arc behavior, Check mode, sign positions, and tow zone rendering
unchanged.

## Data loading

### `app/data-loader.ts`
Add `cleaningSegments` to `StartupStaticData`:

```typescript
export type StartupStaticData = {
  roadGeometry: RoadGeometry | undefined;
  streetParity: Record<string, 1 | -1> | undefined;
  cleaningEntries: StreetCleaningEntry[];
  cleaningSegments: CleaningSegment[];   // NEW — empty if file absent
  snowRoutes: SnowRoute[];
  addressArc: AddressArcIndex | undefined;
};
```

In `loadStartupStaticData`, fetch `data/cleaning-segments.json` in the same `Promise.all`
block as the other static files (non-fatal catch returns `[]`):

```typescript
fetchJson<CleaningSegmentsFile>("data/cleaning-segments.json")
  .then((data) => {
    cleaningSegments = data.segments ?? [];
    console.log(`[cleaning-segments] roadGeometryHash: ${data.roadGeometryHash}`);
  })
  .catch(() => { /* non-fatal — falls back to old inference path */ })
```

Log `roadGeometryHash` at startup for traceability. Do not compute any hash in the browser.

### `app/main.ts`
Pass `cleaningSegments` from `StartupStaticData` into `initMap` / `setCleaningSegments`
(see map.ts changes below). Update tests that mock `loadStartupStaticData` to include
`cleaningSegments: []` in the mock return value.

## `app/map.ts` changes

### 1. Store precomputed segments
Add module-level `let _cleaningSegments: CleaningSegment[] = []`.
Export `function setCleaningSegments(segments: CleaningSegment[]): void` that sets it.

### 2. Feature flag
```typescript
const USE_CLEANING_SEGMENTS = true;
```
This constant can be set to `false` temporarily to compare old vs new rendering output.
When `false`, `renderViolationHighlights` falls through to the existing
`parseLocationBounds`-based inference. Do not use a URL param — a constant is sufficient
for this comparison phase.

### 3. Updated `renderViolationHighlights`

Keep the existing function signature unchanged:
```typescript
export function renderViolationHighlights(
  cleaningEntries: StreetCleaningEntry[],
  now: Date
): void
```

Internal logic when `USE_CLEANING_SEGMENTS && _cleaningSegments.length > 0`:

```
clear existing violation highlight layers

// Group segments by id to find highest-priority status
const byId = new Map<string, { segment: CleaningSegment; status: "active"|"upcoming"|"safe"|"none" }>();
for each segment in _cleaningSegments:
  status = computeCleaningStatus(segment.schedule, now)  // existing status logic
  if status === "none": skip
  existing = byId.get(segment.id)
  if !existing or priority(status) > priority(existing.status):
    byId.set(segment.id, { segment, status })

for each { segment, status } in byId.values():
  color = statusToColor(status)      // existing color logic
  opacity = statusToOpacity(status)  // existing opacity logic

  if segment.geometry is present (no fromArcM):
    ways = [segment.geometry]        // wrap as single way
  else:
    rawWays = _roadGeometry[segment.streetKey] ?? []
    filteredWays = filterWaysByStreetLat(rawWays, segment.streetKey)
    ways = clipWaysToArcRange(filteredWays, segment.fromArcM, segment.toArcM)

  if ways.length === 0: continue
  draw ways with existing offsetPolylinePoints(segment.side) + mergeWays() + buildWayPolylines()
  add resulting polylines to _violationHighlightLayers
```

Priority order: active=3 > upcoming=2 > safe=1 > none=0.

When `USE_CLEANING_SEGMENTS` is `false`, the function falls through to the existing
`cleaningEntries`-based inference unchanged.

### 4. Filtering still applied
`filterWaysByStreetLat` is still called on the arc-clipped ways as a final guard against
any JC geometry that slipped through into the precomputed segments (same pattern as CF-19).

## `package.json` build script

Add to the existing `build` script's `cp` block:
```
&& (cp data/cleaning-segments.json app/data/cleaning-segments.json 2>/dev/null || true)
```

`cleaning-segments.json` is generated by `npm run build-cleaning-segments` (CF-21) and
committed to source. The build cp step copies it to `app/data/` so the browser can fetch it.

## Behavioral contracts

- GIVEN `_cleaningSegments` is populated and `USE_CLEANING_SEGMENTS` is `true`,
  WHEN `renderViolationHighlights` is called with `now` in an active cleaning window for
  segment id "hudson-st-east-14th-st-to-15th-st",
  THEN `clipWaysToArcRange` is called with that segment's `fromArcM`/`toArcM`,
  AND the resulting polyline is added to the violation highlight layers.

- GIVEN a segment with only `geometry: [[40.75, -74.03], [40.751, -74.03]]` and no
  `fromArcM`/`toArcM`,
  WHEN `renderViolationHighlights` is called,
  THEN the segment is rendered from its explicit geometry (no `clipWaysToArcRange` call
  for this segment).

- GIVEN two segments with the same `id` where one has status "active" and one has
  status "safe",
  WHEN `renderViolationHighlights` is called,
  THEN only one polyline is drawn for that id, with the "active" (red) color.

- GIVEN `_cleaningSegments` is empty (cleaning-segments.json absent or empty),
  AND `USE_CLEANING_SEGMENTS` is `true`,
  WHEN `renderViolationHighlights` is called,
  THEN no violation highlights are drawn (no fallthrough to old inference in this state).

- GIVEN `USE_CLEANING_SEGMENTS` is `false`,
  WHEN `renderViolationHighlights` is called,
  THEN it falls through to the existing `parseLocationBounds`-based inference path unchanged.

- GIVEN a safe segment (schedule currently inactive),
  WHEN `renderViolationHighlights` is called,
  THEN the segment renders green.

- GIVEN `loadStartupStaticData` resolves,
  THEN `console.log` includes the `roadGeometryHash` string from the loaded file.

## Test list

1. `renderViolationHighlights` with precomputed segment — calls `clipWaysToArcRange` with
   correct `fromArcM`/`toArcM` (mock `_cleaningSegments` with one segment)
2. Explicit-geometry segment renders without calling `clipWaysToArcRange`
3. Two segments same id: active overrides safe → one red polyline
4. Two segments same id: both safe → one green polyline
5. Empty `_cleaningSegments` with `USE_CLEANING_SEGMENTS=true` → no highlights drawn
6. Safe segment renders green
7. `loadStartupStaticData` populates `cleaningSegments` from fetched JSON
8. `loadStartupStaticData` with missing `cleaning-segments.json` → `cleaningSegments: []`
   (non-fatal)
9. `console.log` called with roadGeometryHash when file loads
