# CF-24 — Street-Cleaning Source Override

## Module
`fetcher/fetch-street-cleaning.ts` (modified), `shared/types.ts` (modified),
`data/street-cleaning-overrides.json` (new), `tests/unit/fetch-street-cleaning.test.ts` (modified)

## Problem
The Hoboken street-cleaning schedule page currently lists Monroe St. (both sides) as cleaning
between "Eleventh St. to Newark St." Posted signs and road geometry confirm the zone extends
to 12th St. Every re-scrape of the city page writes the wrong endpoint back to
`data/street-cleaning.json`, and CF-21's segment generator inherits the bad `fromLabel:
"11TH ST"` for both Monroe segments.

There is no mechanism to override known-bad municipal rows at scrape time. Corrections applied
downstream (e.g. in `cleaning-segment-overrides.json`) are fragile because the segment IDs are
derived from the scraped location strings — a bad source row produces a bad segment ID.

## Goal
Add a committed override layer applied immediately after `parseCleaningHtml` in `runScraper`.
The override file (`data/street-cleaning-overrides.json`) is read at each scrape, matched
against parsed entries using normalized text, and patched entries replace the originals before
the output JSON is written.

## New type — `shared/types.ts`

Add after the `StreetCleaningData` interface:

```typescript
export interface StreetCleaningEntryOverride {
  match: {
    street: string;   // human-readable scraped value; matched after normalization
    side: string;     // human-readable scraped value; matched after normalization
    location: string; // human-readable scraped value; matched after normalization
  };
  replace: Partial<Pick<StreetCleaningEntry, "location" | "schedule">>;
  reason: string;
}
```

## New functions — `fetcher/fetch-street-cleaning.ts`

Import `StreetCleaningEntryOverride` on the existing import line from `../shared/types`.

### `normalizeOverrideText(s: string): string` (internal, not exported)

Canonicalizes text for fuzzy matching. Handles spelled-out ordinals vs. numerics, missing
periods, and "Street" vs. "St.":

```typescript
function normalizeOverrideText(s: string): string {
  return s
    .toLowerCase()
    .replace(/\./g, "")
    .replace(/\bstreet\b/g, "st")
    .replace(/\beleventh\b/g, "11th")
    .replace(/\btwelfth\b/g, "12th")
    .replace(/\s+/g, " ")
    .trim();
}
```

### `applyStreetCleaningOverrides(entries, overrides)` (exported)

Pure function — no logging, no I/O. Returns patched entries and log lines for the caller:

```typescript
export function applyStreetCleaningOverrides(
  entries: StreetCleaningEntry[],
  overrides: readonly StreetCleaningEntryOverride[]
): { entries: StreetCleaningEntry[]; applied: string[] }
```

Matching: for each entry, find all overrides where normalized `match.street`, `match.side`,
and `match.location` all equal the normalized entry fields. Rules:
- 0 matches → return entry unchanged
- 2+ matches → throw `Error` with message containing "remove the duplicate" (ASCII hyphens, no em dashes)
- 1 match → apply `override.replace` fields onto the entry, push a log line:
  `[street-cleaning override] ${street} ${side}: "${oldLocation}" -> "${newLocation}"`

Do not call `console.log` inside this function.

### `loadStreetCleaningOverrides(fs, dataDir)` (internal, not exported)

Reads and validates `data/street-cleaning-overrides.json`. Returns `[]` if the file does not
exist. Throws if the file contains invalid JSON or if the parsed value is not an array:

```typescript
function loadStreetCleaningOverrides(
  fs: FsBackend,
  dataDir: string
): StreetCleaningEntryOverride[] {
  const p = nodePath.join(dataDir, "street-cleaning-overrides.json");
  if (!fs.existsSync(p)) return [];
  const parsed = JSON.parse(fs.readFileSync(p, "utf8")) as unknown;
  if (!Array.isArray(parsed)) {
    throw new Error("street-cleaning-overrides.json must contain an array");
  }
  return parsed as StreetCleaningEntryOverride[];
}
```

### Wire into `runScraper`

Replace the current `entries` usage with:

```typescript
const entries = parseCleaningHtml(html);
const overrides = loadStreetCleaningOverrides(fs, DATA_DIR);
const { entries: patched, applied } = applyStreetCleaningOverrides(entries, overrides);
for (const line of applied) console.log(line);
const output: StreetCleaningData = { fetched_at: new Date().toISOString(), entries: patched };
```

## New data file — `data/street-cleaning-overrides.json`

Create this file with the Monroe St. corrections. Use ASCII hyphens in reason strings:

```json
[
  {
    "match": {
      "street": "Monroe St.",
      "side": "West",
      "location": "Eleventh St. to Newark St."
    },
    "replace": {
      "location": "Twelfth St. to Newark St."
    },
    "reason": "Posted Monroe St. signs and road geometry show the Monday 1-2 pm cleaning zone extends to 12th St.; Hoboken website currently lists 11th St."

    "match": {
```

## Tests — `tests/unit/fetch-street-cleaning.test.ts`

Add a new `describe("applyStreetCleaningOverrides", ...)` block. Use inline fixtures (no HTML
parsing needed — construct `StreetCleaningEntry` objects directly).

### Given/When/Then

**Test 1 — Monroe West (spelled-out ordinal)**
- Given: one entry `{ street: "Monroe St.", side: "West", schedule: "Monday   1 pm - 2 pm", location: "Eleventh St. to Newark St." }`
- And: one override matching that entry with `replace: { location: "Twelfth St. to Newark St." }`
- When: `applyStreetCleaningOverrides([entry], [override])`
- Then: returned `entries[0].location === "Twelfth St. to Newark St."`
- And: `applied` has length 1 and `applied[0]` contains "Twelfth St. to Newark St."

**Test 2 — Monroe East (spelled-out ordinal)**
- Same as Test 1 but `side: "East"` and schedule contains "Tuesday"
- Then: override is applied, `applied` has one entry

**Test 3 — numeric variant triggers the same override**
- Given: entry with `location: "11th St. to Newark St."` (numeric, no period after "St")
- And: the same override with `match.location: "Eleventh St. to Newark St."` (spelled-out)
- When: `applyStreetCleaningOverrides([entry], [override])`
- Then: `entries[0].location === "Twelfth St. to Newark St."` (normalizer collapses both forms)

**Test 4 — idempotent when Hoboken corrects their data**
- Given: entry with `location: "Twelfth St. to Newark St."` (already correct)
- And: the standard Monroe override (matching "Eleventh St.")
- When: `applyStreetCleaningOverrides([entry], [override])`
- Then: `applied` is empty, `entries.length === 1`, `entries[0].location === "Twelfth St. to Newark St."`

**Test 5 — non-Monroe entry with "Eleventh St." in location is untouched**
- Given: entry `{ street: "Adams St.", side: "West", ..., location: "Eleventh St. to Newark St." }`
- And: the Monroe override (street: "Monroe St.")
- When: `applyStreetCleaningOverrides([entry], [override])`
- Then: `applied` is empty, `entries[0].location === "Eleventh St. to Newark St."`

**Test 6 — duplicate overrides throw**
- Given: one entry and two overrides that both match it
- When: `applyStreetCleaningOverrides([entry], [override1, override2])`
- Then: throws an `Error` whose message contains "remove the duplicate"
