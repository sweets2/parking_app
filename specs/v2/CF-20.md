# CF-20 — Extract Shared Cleaning Geometry Utilities

## Module
`shared/cleaning-geometry.ts` (new), `app/map.ts` (modified)

## Problem
The geometry utilities `normalizeToGeometryKey`, `filterWaysByStreetLat`,
`flattenWaysToArcPath`, `clipWaysToArcRange`, and their associated constants
(`HOBOKEN_STREET_LAT`, `HOBOKEN_STREET_LAT_TOLERANCE`, `HOBOKEN_STREET_LAT_PER_STREET_TOL`)
are embedded in `app/map.ts` alongside Leaflet-dependent rendering code. The upcoming
build-time cleaning segment generator (`scripts/build-cleaning-segments.ts`) needs these
utilities in a Node.js context — it cannot import `app/map.ts` because that module has
Leaflet dependencies. This feature extracts the pure geometry utilities into
`shared/cleaning-geometry.ts`, which has no browser globals, no Leaflet, and no I/O.

## Goal
Pure refactor — no behavior change. Extract pure geometry utilities from `app/map.ts` into
`shared/cleaning-geometry.ts`. `app/map.ts` imports them back. All existing tests pass
with identical results.

**Success criterion**: `npm test` output is identical before and after. No new tests are
required — regression is the only measure. Do not add tests, do not change rendering logic,
do not change data loading.

## What to move to `shared/cleaning-geometry.ts`

All of these are pure functions/constants with no Leaflet dependency:

```typescript
// Move as-is, make all exported:
export function normalizeToGeometryKey(name: string): string
export const HOBOKEN_STREET_LAT: Record<string, number>
export const HOBOKEN_STREET_LAT_TOLERANCE: number
export const HOBOKEN_STREET_LAT_PER_STREET_TOL: Partial<Record<string, number>>
export function filterWaysByStreetLat(
  ways: [number, number][][],
  streetKey: string
): [number, number][][]
export function flattenWaysToArcPath(
  ways: [number, number][][]
): { points: [number, number][]; cumArc: number[]; wayEnds: number[] }
export function clipWaysToArcRange(
  ways: [number, number][][],
  fromArcM: number,
  toArcM: number
): [number, number][][]
```

`shared/cleaning-geometry.ts` must not import any Leaflet types, DOM globals, or Node.js
built-ins. It may import type-only from `shared/types.ts` if needed.

## What stays in `app/map.ts`

Everything else. In particular: all rendering functions, `getCrossStreetCoord`,
`parseLocationBounds`, `clipWaysToBounds`, `resolveCheckCurbWays`, `renderViolationHighlights`,
and all `L.*` Leaflet usage stay in `app/map.ts`.

## How to update `app/map.ts`

1. Delete the definitions of all 7 symbols listed above from `app/map.ts`.
2. Add an import from `../shared/cleaning-geometry` for all 7.
3. `flattenWaysToArcPath` and `clipWaysToArcRange` are currently exported from `app/map.ts`.
   Check whether any file other than `app/map.ts` imports them from `app/map.ts` (grep
   `from.*app/map` in tests/). If any test imports them directly from `app/map`, keep
   a re-export: `export { flattenWaysToArcPath, clipWaysToArcRange } from "../shared/cleaning-geometry"`.
   If no test does, no re-export needed.
4. The comment `// CRITICAL: This function must be IDENTICAL to flattenWaysToArcPath in`
   currently above `flattenWaysToArcPath` in `map.ts` documents a cross-module contract.
   Move this comment (updated to say "shared/cleaning-geometry.ts and build-time usage") to
   `shared/cleaning-geometry.ts` above `flattenWaysToArcPath`.

## Behavioral contracts

- GIVEN any call to normalizeToGeometryKey, filterWaysByStreetLat, flattenWaysToArcPath,
  or clipWaysToArcRange from map.ts before this change,
  WHEN the same call is made after this change (now via shared import),
  THEN the return value is bit-for-bit identical.

- GIVEN `npm test` passes before this change,
  WHEN this change is applied,
  THEN `npm test` passes with the same test count and no new failures.

No new tests. The existing test suite is the verification.
