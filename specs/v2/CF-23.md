# CF-23 — Remove Old Runtime Geometry Inference

## Module
`app/map.ts`, `tests/unit/map.test.ts`

## Problem
CF-22 added `USE_CLEANING_SEGMENTS = true` and left the old `parseLocationBounds`-based
inference path in `renderViolationHighlights` behind a flag. The precomputed segment path
is now live. The old runtime inference code is dead weight: it makes the code harder to
read and tests for it test a path that no longer runs. This feature removes the dead path
and the tests that covered it.

## Goal
Remove the `USE_CLEANING_SEGMENTS` flag and the old `StreetCleaningEntry[]`-based inference
path from `renderViolationHighlights`. Clean up any functions that are now exclusively
called by the removed path. Update tests.

## Changes to `app/map.ts`

### `renderViolationHighlights`
Remove:
- `const USE_CLEANING_SEGMENTS = true` constant
- The `if (USE_CLEANING_SEGMENTS && _cleaningSegments.length > 0)` branch guard — the
  precomputed segment path becomes the only path (no guard needed; if `_cleaningSegments`
  is empty, no highlights draw)
- The old inference block (the `else` path that called `parseLocationBounds`,
  `getCrossStreetCoord`, `clipWaysToBounds` in a loop over `cleaningEntries`)

After this change, `renderViolationHighlights` iterates `_cleaningSegments` only. The
`cleaningEntries: StreetCleaningEntry[]` parameter can be removed from the function
signature if it is no longer used inside the function. If it is still passed from
`app/main.ts` for another purpose, keep the parameter but mark it unused or remove the
call site.

### `parseLocationBounds`
Check if `parseLocationBounds` is called anywhere other than the removed inference block:
- `resolveCheckCurbWays` — YES, it calls `parseLocationBounds`. Do NOT remove it.
- Any other callers? Grep for `parseLocationBounds` in the codebase.
- If `parseLocationBounds` is still called by at least one live function, leave it in
  `app/map.ts` unchanged.
- If it is now unused, remove it.

### `clipWaysToBounds`
Check if `clipWaysToBounds` is called anywhere other than the removed inference block:
- Grep for `clipWaysToBounds` in the codebase.
- If unused after the removal, delete it.
- If still called, leave it.

### `getCrossStreetCoord`
Check if `getCrossStreetCoord` is called anywhere other than the removed inference block:
- Grep for `getCrossStreetCoord` in the codebase.
- If unused, delete it.
- If still called (e.g. by `resolveCheckCurbWays`), leave it.

## Changes to `tests/unit/map.test.ts`

### Tests to remove
The following tests tested the old runtime inference path exclusively — they set up
synthetic `StreetCleaningEntry[]` and checked that `renderViolationHighlights` filtered
JC geometry correctly. These are no longer valid tests because `renderViolationHighlights`
no longer infers geometry from entries:

- All unit tests added by CF-17 (2nd Street safe highlights / parseLocationBounds fallthrough)
  that test the rendering path via raw entries
- All unit tests added by CF-18 (Madison/Monroe contamination) that test the rendering path
  via raw entries
- All unit tests added by CF-19 (per-street centroid filter) that test `filterWaysByStreetLat`
  being applied inside `renderViolationHighlights` via raw entries

Before removing, verify each test: if a test exercises `filterWaysByStreetLat` in isolation
(not through `renderViolationHighlights`), keep it. If it exercises the old inference path
end-to-end, remove it.

### Tests to keep
- CF-19 real-data audit test (loads actual data files, runs `renderViolationHighlights`
  with `_cleaningSegments` populated from cleaning-segments.json, asserts no JC geometry
  in rendered output) — this tests the final guarantee and is still valid. Update it to
  load and set `_cleaningSegments` instead of passing raw `cleaningEntries`.
- All CF-22 tests for the precomputed path
- All tests for `resolveCheckCurbWays`, Check mode, address arc, sign positions

### Tests to add
- GIVEN `renderViolationHighlights` is called after CF-23 (no USE_CLEANING_SEGMENTS flag),
  WHEN `_cleaningSegments` is empty,
  THEN no polylines are added and no error is thrown.

## Behavioral contracts

- GIVEN the `USE_CLEANING_SEGMENTS` flag is removed,
  WHEN `renderViolationHighlights` is called with any `cleaningEntries` argument,
  THEN it renders from `_cleaningSegments` only, regardless of what is in `cleaningEntries`.

- GIVEN `_cleaningSegments` is empty,
  WHEN `renderViolationHighlights` is called,
  THEN no violation highlights are drawn and no error is thrown.

- GIVEN `resolveCheckCurbWays` exists and calls `parseLocationBounds`,
  WHEN this feature runs,
  THEN `parseLocationBounds` and `getCrossStreetCoord` are retained (not removed).

- GIVEN `npm test` passed after CF-22,
  WHEN the old-path tests are removed and the inference code is deleted,
  THEN `npm test` still passes with the remaining tests.
