# CF-19 — Systemic Street-Cleaning Geometry Fix: Per-Street Centroid Filter

## Module
`app/map.ts` — `filterWaysByStreetLat`, `parseLocationBounds`, `renderViolationHighlights`, `resolveCheckCurbWays`

## Problem
`road-geometry.json` was built from an OSM bbox larger than Hoboken. Numbered streets
contain both Hoboken ways and Jersey City ways in two contamination clusters:

- **South JC cluster** (9TH–16TH ST): ways at lat 40.728–40.733. These are INSIDE the
  Hoboken lat window (40.728–40.760), so the rectangular `HOBOKEN_LAT_MIN/MAX` fallback
  in `renderViolationHighlights` does NOT filter them.
- **North JC cluster / JC Heights** (2ND–13TH ST): ways at lat 40.753–0.760. These
  inflate `latSpread` in `parseLocationBounds`, causing the N-S → E-W fallthrough to
  misfire and `getBoundaryCoord` to return wrong boundary coordinates.

### Bug 1 — `parseLocationBounds` uses raw unmasked geometry
`lats`/`lngs` are computed from ALL ways including JC. `getBoundaryCoord` (west/east
boundary terms) returns the westernmost lon of every OSM way with that name. The
`latSpread`/`lngSpread` orientation test can be inflated by JC ways.

### Bug 2 — `renderViolationHighlights` fallback is lat-only
When `parseLocationBounds` returns null, the fallback `clipWaysToBounds(..., "lat")`
clips only by `HOBOKEN_LAT_MIN/MAX`. This leaves south JC ways (at valid latitudes like
40.729) and north-boundary contamination through.

## Fix
Expand `HOBOKEN_STREET_LAT` to all 16 numbered streets with calibrated centroid values.
Add `filterWaysByStreetLat(ways, streetKey)` that drops whole JC ways before they can
affect boundary computations or rendered output. Apply it in `parseLocationBounds` (so
`getBoundaryCoord` uses clean geometry), `renderViolationHighlights` (final guard after
clip), and `resolveCheckCurbWays` (so check-mode stripes also use clean geometry).

For streets 11TH, 12TH, and 13TH ST the gap between the nearest JC centroid and the
Hoboken cluster centroid is ≤ 0.008°. A per-street tight tolerance map
(`HOBOKEN_STREET_LAT_PER_STREET_TOL`) provides overrides: 11TH ST → 0.007, 12TH ST
→ 0.006, 13TH ST → 0.007.

## New helper

```typescript
const HOBOKEN_STREET_LAT_PER_STREET_TOL: Partial<Record<string, number>> = {
  "11TH ST": 0.007,
  "12TH ST": 0.006,
  "13TH ST": 0.007,
};

function filterWaysByStreetLat(
  ways: [number, number][][],
  streetKey: string,
): [number, number][][] {
  const expected = HOBOKEN_STREET_LAT[streetKey];
  if (expected === undefined) return ways; // non-numbered: pass through
  const tol = HOBOKEN_STREET_LAT_PER_STREET_TOL[streetKey] ?? HOBOKEN_STREET_LAT_TOLERANCE;
  return ways.filter(way => {
    if (way.length === 0) return false;
    const centLat = way.reduce((sum, pt) => sum + pt[0], 0) / way.length;
    return Math.abs(centLat - expected) <= tol;
  });
}
```

## Patches

### parseLocationBounds (Step 3)
Before computing `lats`/`lngs`:
```typescript
// Before:
const ways = _roadGeometry[streetKey];
if (ways === undefined || ways.length === 0) return null;
const allPts = ways.flat();

// After:
const rawWays = _roadGeometry[streetKey];
if (rawWays === undefined || rawWays.length === 0) return null;
const ways = filterWaysByStreetLat(rawWays, streetKey);
if (ways.length === 0) return null;
const allPts = ways.flat();
```

### renderViolationHighlights (Step 4)
After existing clip:
```typescript
// Before:
const ways = bounds !== null
  ? clipWaysToBounds(allWays, bounds.min, bounds.max, bounds.axis)
  : clipWaysToBounds(allWays, HOBOKEN_LAT_MIN, HOBOKEN_LAT_MAX, "lat");
if (ways.length === 0) continue;

// After:
const clipped = bounds !== null
  ? clipWaysToBounds(allWays, bounds.min, bounds.max, bounds.axis)
  : clipWaysToBounds(allWays, HOBOKEN_LAT_MIN, HOBOKEN_LAT_MAX, "lat");
const ways = filterWaysByStreetLat(clipped, street);
if (ways.length === 0) continue;
```

### resolveCheckCurbWays (Step 4b)
Apply filter to rawWays before bounds computation:
```typescript
// Before:
const allWays = _roadGeometry[streetKey];
if (allWays === undefined || allWays.length === 0) return null;
const bounds = parseLocationBounds(location, streetKey);

// After:
const rawWays = _roadGeometry[streetKey];
if (rawWays === undefined || rawWays.length === 0) return null;
const allWays = filterWaysByStreetLat(rawWays, streetKey);
if (allWays.length === 0) return null;
const bounds = parseLocationBounds(location, streetKey);
```

Note: `parseLocationBounds` now also filters internally (Step 3), so boundary coords
are already computed from clean geometry. `resolveCheckCurbWays` filters independently
to ensure check-mode stripe rendering also uses clean ways.

## Behavioral contracts

- GIVEN 9TH ST ways include a Hoboken way (avgCentLat 40.748) and a south JC way
  (avgCentLat 40.729) and a JC Heights way (avgCentLat 40.757),
  WHEN `renderViolationHighlights` is called,
  THEN no rendered point has lat < 40.740 AND no rendered point has lat > 40.756,
  AND the Hoboken way still renders.

- GIVEN 12TH ST ways include Hoboken ways (avgCentLat ~40.752) and a south JC way
  (avgCentLat 40.731) and a JC Heights way (avgCentLat 40.759),
  WHEN `renderViolationHighlights` is called,
  THEN no rendered point has avgCentLat below 40.740 or above 40.760.

- GIVEN 5TH ST ways include a Hoboken way (avgCentLat 40.742) and a JC Heights way
  (avgCentLat 40.758),
  WHEN `renderViolationHighlights` is called,
  THEN no rendered point has lat > 40.750, AND the Hoboken way still renders.

- GIVEN 13TH ST ways include Hoboken (40.753), south JC (40.730), and north JC (40.761),
  WHEN `renderViolationHighlights` is called,
  THEN both contaminated ways are dropped and the Hoboken way renders.

- GIVEN 12TH ST raw ways include a south JC way at avgCentLat 40.730,
  WHEN `parseLocationBounds` is called with location "West boundary to Willow Ave.",
  THEN the "WEST BOUNDARY" lng is derived from masked ways only.

- GIVEN the CF-17 2ND ST fallthrough scenario (MARSHALL ST distance > 0.005°),
  THEN the centroid filter does not affect the fallthrough — MARSHALL ST still fails
  the distance check, so fallthrough still triggers.

- GIVEN a non-numbered street (e.g. GROVE ST) with contaminated geometry,
  WHEN `filterWaysByStreetLat` is called,
  THEN all ways are returned unchanged (no filter applied for non-numbered streets).
