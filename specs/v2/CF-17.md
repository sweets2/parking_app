# CF-17 — Fix 2nd Street Safe Highlights (parseLocationBounds Fallthrough)

## Module
`app/map.ts` — `parseLocationBounds`

## Problem
`parseLocationBounds` uses max-min spread to classify streets as N-S or E-W. For "2ND ST",
3 Jersey City Heights contamination ways inflate latSpread (0.015865°) above lngSpread
(0.015550°), causing wrong N-S classification. In the N-S branch, cross-street lookups
for MARSHALL ST and RIVER ST fail (too far from median longitude) → function returns null
→ fallback renders 2ND ST without location clipping → no visible green safe highlights.

## Fix
Change the strict N-S/E-W if/else in `parseLocationBounds` to a fallthrough: if either
cross-street lookup fails in the N-S branch, attempt the E-W interpretation before
returning null.

## Behavioral contract
- GIVEN a street whose geometry is contaminated with JC Heights ways such that
  latSpread ≥ lngSpread, but whose actual Hoboken portion is E-W,
  WHEN its cross-streets are findable near the median latitude,
  THEN `parseLocationBounds` returns `{ axis: "lng", … }` via the fallthrough path.

- GIVEN a genuine N-S street (latSpread >> lngSpread),
  WHEN its E-W cross-streets are near its median longitude,
  THEN `parseLocationBounds` returns `{ axis: "lat", … }` via the N-S branch without
  reaching the fallthrough.

- GIVEN an E-W street where lngSpread > latSpread (5TH–8TH ST pattern with heavy
  JC contamination),
  THEN `parseLocationBounds` enters the E-W branch directly, fallthrough never triggered.

## Test list
- GIVEN 2ND ST with JC contamination + cross-street geometry, safe schedule at 8 PM ET → green polylines on map
- GIVEN 2ND ST Willow-to-Marshall location → rendered lons within [−74.048, −74.029], River-side way excluded
- GIVEN E-W street with dominant JC contamination (7 JC / 4 Hoboken) → red polylines rendered (no regression)
- GIVEN long N-S street + cross-street geometry → red polylines rendered (N-S branch succeeds, no fallthrough)
