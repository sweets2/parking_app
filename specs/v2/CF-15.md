# CF-15 — Static Data Files

## What this is

Two static JSON data files bundled with the app: municipal parking garages and snow emergency routes. These are hand-curated, not scraped — they are committed to the repo.

## Output files

- `data/garages.json`
- `data/snow-emergency-routes.json`
- `tests/unit/static-data.test.ts`

## `data/garages.json`

A JSON array of `Garage` objects (type from `shared/types.ts`):

```typescript
interface Garage {
  name: string;       // e.g. "Garage B"
  address: string;    // e.g. "28 2nd St"
  capacity: number;   // integer parking spaces
  lat: number;
  lng: number;
  phone: string;      // e.g. "201-653-7333"
}
```

Must include all Hoboken municipal parking garages. At minimum:
- Southwest Garage (14th St & Grand St area)
- 2nd Street Garage (Garage B)
- 4th Street Garage (Midtown Garage, near Observer Hwy)
- Newark St / Hudson St area garage (Garage D or G)

Each entry must have accurate Hoboken coordinates (lat ~40.74–40.75, lng ~-74.04 to -74.02).

## `data/snow-emergency-routes.json`

A JSON object with a `routes` array of `SnowRoute` objects (type from `shared/types.ts`):

```json
{
  "routes": [ ... ]
}
```

Each `SnowRoute` entry:

```typescript
interface SnowRoute {
  street: string;   // road-geometry key, e.g. "3RD ST" (uppercase, no periods)
  side: string;     // "North" | "South" | "Both" | "West" | "East"
  from: string;     // cross-street label for display only
  to: string;       // cross-street label for display only
  minLon?: number;  // clip geometry west of this longitude (east-west streets)
  maxLon?: number;  // clip geometry east of this longitude (east-west streets)
  minLat?: number;  // clip geometry south of this latitude (north-south streets)
  maxLat?: number;  // clip geometry north of this latitude (north-south streets)
}
```

Must cover Hoboken's designated snow emergency routes. The `street` values must match the normalized road geometry keys (e.g., `"WASHINGTON ST"`, `"HUDSON ST"`, `"OBSERVER HWY"`).

### Known constraints on snow route data

1. **Centroid-lat filtering** — `renderSnowEmergencyRoutes` in `app/map.ts` filters out ways whose centroid falls outside `HOBOKEN_STREET_LAT[street] ± HOBOKEN_STREET_LAT_TOLERANCE`. Snow route data for numbered streets must use the correct expected latitude values:
   - `"9TH ST"`: expected lat 40.748 (not 40.750)
   - `"13TH ST"`: expected lat 40.753 (not 40.756)

2. **Per-route clipping** — Use `minLon`/`maxLon` for east-west streets (numbered streets — clip to the cross streets' longitude). Use `minLat`/`maxLat` for north-south streets (e.g., `WILLOW AVE` uses `maxLat: 40.752` to stop at 11th St).

3. **Build script cp block** — Both `data/garages.json` and `data/snow-emergency-routes.json` must be included in the `cp` block of the `build` script in `package.json` so the browser fetch can reach them at `app/data/`.

## Test file: `tests/unit/static-data.test.ts`

Uses vitest. Reads both JSON files from disk (Node `fs.readFileSync`) and validates their structure. No TypeScript compilation of the JSON — only structural assertions.

## Test cases

### garages.json structural tests

**GIVEN** `data/garages.json` is read and parsed as JSON  
**WHEN** the result is checked  
**THEN** it is an array with at least 3 entries

**GIVEN** each entry in `data/garages.json`  
**WHEN** its fields are inspected  
**THEN** `name` is a non-empty string, `address` is a non-empty string, `capacity` is a positive integer, `lat` is between 40.72 and 40.77, `lng` is between -74.06 and -74.01, `phone` is a non-empty string

**GIVEN** `data/garages.json`  
**WHEN** garage names are listed  
**THEN** at least one entry has a name matching `/garage/i` or `/midtown/i` or `/southwest/i`

### snow-emergency-routes.json structural tests

**GIVEN** `data/snow-emergency-routes.json` is read and parsed as JSON  
**WHEN** the result is checked  
**THEN** it is an object with a `routes` property that is an array with at least 4 entries

**GIVEN** each entry in the `routes` array of `data/snow-emergency-routes.json`  
**WHEN** its fields are inspected  
**THEN** `street` is a non-empty uppercase string with no periods, `side` is one of `"North"`, `"South"`, `"Both"`, `"West"`, `"East"`, `from` is a non-empty string, `to` is a non-empty string

**GIVEN** `data/snow-emergency-routes.json`  
**WHEN** street names in `routes` are listed  
**THEN** at least one entry has `street` equal to `"OBSERVER HWY"` or `"WASHINGTON ST"` or `"WILLOW AVE"`

**GIVEN** `data/snow-emergency-routes.json`  
**WHEN** any entry has `minLon`, `maxLon`, `minLat`, or `maxLat`  
**THEN** those values are numbers in the range of valid Hoboken coordinates (lon -74.06 to -74.01, lat 40.72 to 40.77)

## Hard constraints

- `data/garages.json` must be a valid JSON array of `Garage` objects — no wrapper object.
- `data/snow-emergency-routes.json` must be a valid JSON object with a `routes` key containing an array of `SnowRoute` objects.
- Coordinates must be real Hoboken coordinates (not placeholders).
- `street` values in snow route entries must use the normalized geometry key format: uppercase, abbreviated, no periods (e.g. `"3RD ST"` not `"Third Street"`).
- Both files must be valid JSON.
- No `as any`. No `!` non-null assertions. All test exports use named exports.
