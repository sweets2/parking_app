# CF-14 — Analytics and Feedback

## What this is

Two small feature modules:

1. `app/analytics.ts` — lightweight analytics tracking (Umami).
2. `app/feedback.ts` — floating feedback button and modal.

## Output files

- `app/analytics.ts`
- `app/feedback.ts`
- `api/feedback.ts`
- `tests/unit/analytics.test.ts`
- `tests/unit/feedback.test.ts`

---

## `app/analytics.ts`

```typescript
export function track(event: string, data?: Record<string, unknown>): void
// Sends an event to the Umami analytics instance.
// Guards with `typeof window !== "undefined"` — safe to call in Node tests.
// Uses window.umami?.track if available; silently no-ops otherwise.
```

No module-level side effects. No initialization required.

### Test list (`tests/unit/analytics.test.ts`)

1. `track` does not throw in a Node environment (no window)
2. `track` calls `window.umami.track` when umami is available
3. `track` does not throw when `window.umami` is undefined

---

## `app/feedback.ts`

```typescript
export function initFeedback(): void
// Wires the feedback button (#feedback-btn) to open/close the feedback modal (#feedback-overlay).
// The modal contains a form that POSTs to /api/feedback.
// On successful submit, closes the modal and shows a confirmation message.
```

The feedback modal is shown when `#feedback-btn` is clicked and hidden when:
- The user clicks the `×` close button inside the modal
- The user clicks the overlay backdrop (`#feedback-overlay`) outside the modal
- The form is successfully submitted

### Test list (`tests/unit/feedback.test.ts`)

1. `initFeedback` does not throw when feedback elements are absent from DOM
2. Clicking `#feedback-btn` sets `#feedback-overlay` hidden=false
3. Clicking the overlay backdrop (`#feedback-overlay` directly, not the modal) hides the overlay
4. `initFeedback` wires the close button inside the modal

---

## `api/feedback.ts`

A Vercel serverless function that receives feedback form POSTs.

```typescript
// Vercel Edge/Node API route
// POST /api/feedback
// Body: { message: string, email?: string }
// Stores or forwards the feedback (implementation may be a stub that returns 200 OK).
```

The exact storage mechanism (email, database, etc.) is implementation-defined. A minimal implementation that always returns `{ ok: true }` is acceptable.

## Hard constraints

- No `as any`. No `!`. Named exports only.
- `track` must guard with `typeof window !== "undefined"`.
- `initFeedback` must not throw when DOM elements are missing.
- Tests run in `environment: "node"` — no browser globals without explicit fake DOM setup.
