## `app/feedback.ts`

```typescript
export function initFeedback(): void
// Wires the feedback button (#feedback-btn) to open/close the feedback modal (#feedback-overlay).
// The modal contains a form that POSTs to /api/feedback.
// On successful submit, closes the modal and shows a confirmation message.
```

The feedback modal is shown when `#feedback-btn` is clicked and hidden when:
- The user clicks the `×` close button inside the modal
- The user clicks the overlay backdrop (`#feedback-overlay`) outside the modal
- The form is successfully submitted

On successful form submit: hide `#feedback-overlay` (set `hidden` to `true`) and show `#feedback-confirm` (remove the `hidden` attribute). The `#feedback-confirm` element is a `<div>` that is hidden by default via the `hidden` HTML attribute. Removing the `hidden` attribute makes it visible. No text-content change is required on `#feedback-confirm` — the static HTML already contains the confirmation text.

### Test list (`tests/unit/feedback.test.ts`)

**Given** a DOM with no `#feedback-btn` or `#feedback-overlay` elements,
**When** `initFeedback()` is called,
**Then** it does not throw.

**Given** a DOM containing `#feedback-btn` and `#feedback-overlay` (hidden by default),
**When** a click event is dispatched on `#feedback-btn`,
**Then** `#feedback-overlay` is no longer hidden (its `hidden` attribute is `false` or removed).

**Given** a DOM with a visible `#feedback-overlay`,
**When** a click event is dispatched directly on `#feedback-overlay` (not on a child element),
**Then** `#feedback-overlay` is hidden.

**Given** a DOM with `#feedback-btn`, `#feedback-overlay`, and a close button inside the modal,
**When** `initFeedback()` is called and the close button is clicked,
**Then** `#feedback-overlay` is hidden.

**Given** a DOM with `#feedback-btn`, `#feedback-overlay`, a `<form>` element inside the overlay, a `<textarea name="message">` with value `"great app"`, and a `<div id="feedback-confirm">` element (initially hidden via `hidden` attribute),
**When** `initFeedback()` is called and the form's `submit` event is fired (with `preventDefault` called),
**Then** `#feedback-overlay` is hidden and `#feedback-confirm` has its `hidden` attribute removed (i.e. `hidden` is `false` or the attribute is absent).

## `api/feedback.ts`

A Vercel Edge serverless function that receives feedback form POSTs.

```typescript
export const config: { runtime: string }
// Vercel Edge config — set to { runtime: "edge" }

export async function handler(req: Request): Promise<Response>
// POST /api/feedback
// Reads body as JSON: { message: string, email?: string }
// Returns 405 if method is not POST.
// Returns 400 JSON { error: "message required" } if message is missing or empty.
// Returns 200 JSON { ok: true } if the message is accepted.
// Any email/database forwarding is implementation-defined; a stub that skips
// forwarding and always returns 200 { ok: true } when message is non-empty is acceptable.
```

> Note: The handler is exported as a **named export** (`export async function handler`) in compliance with CLAUDE.md's no-default-exports rule. The Vercel config object (`export const config`) signals the Edge runtime. No default export is used or needed.

### Test list (`tests/unit/api-feedback.test.ts`)

**Given** a POST request with body `{ "message": "hello" }`,
**When** `handler(req)` is called,
**Then** the response status is `200` and the JSON body equals `{ ok: true }`.

**Given** a POST request with body `{ "message": "" }` (empty string),
**When** `handler(req)` is called,
**Then** the response status is `400` and the JSON body contains `{ error: "message required" }`.

**Given** a POST request with body `{}` (no message field),
**When** `handler(req)` is called,
**Then** the response status is `400` and the JSON body contains `{ error: "message required" }`.

**Given** a GET request,
**When** `handler(req)` is called,
**Then** the response status is `405`.

> Note: `tests/unit/api-feedback.test.ts` is the test file for `api/feedback.ts`. It runs in Node (`environment: "node"`), constructs synthetic `Request` objects directly using `new Request(...)` (available in Node 18+ via the Web API), and does not require a DOM.

## `tests/unit/feedback.test.ts` — environment note

`tests/unit/feedback.test.ts` tests `app/feedback.ts`, which calls `document.getElementById`. Because the global vitest config sets `environment: "node"`, no real DOM is available. The test file **must** install a fake `document` object on `globalThis` in a `beforeEach` block — using an in-memory map of fake elements — before calling `initFeedback()`. Each fake element tracks `classList` membership, `addEventListener` callbacks, attribute values, `value`, `textContent`, `className`, and `disabled` fields. Event handlers are fired by calling the stored listeners directly. The `afterEach` block must `delete (globalThis as Record<string, unknown>).document` to clean up. No `jsdom` or `happy-dom` dependency is needed.

## Hard constraints

- No `as any`. No `!`. Named exports only. No default exports anywhere in this feature.
- `track` must guard with `typeof window !== "undefined"`.
- `initFeedback` must not throw when DOM elements are missing.
- Tests run in `environment: "node"` — no browser globals without explicit fake DOM setup as described above.
