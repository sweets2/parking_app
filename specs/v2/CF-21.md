# CF-21 — Build-Time Cleaning Segment Generator

## Module
`scripts/build-cleaning-segments.ts` (new), `data/cleaning-segment-overrides.json` (new),
`tests/unit/build-cleaning-segments.test.ts` (new), `package.json` (modified)

## Problem
`renderViolationHighlights` currently infers cleaning segment geometry live in the browser:
`parseLocationBounds` → `getCrossStreetCoord` → `clipWaysToBounds`. This is fragile —
each fix to handle one street breaks another. The correct approach is to compute cleaning
segment geometry once at build time, audit every output, and let the browser render
pre-verified data.

## Goal
Create a build-time script that:
1. Reads `data/street-cleaning.json`, `data/road-geometry.json`, and
   `data/cleaning-segment-overrides.json`.
2. For each cleaning row, resolves the segment's location to arc meter positions
   (`fromArcM`/`toArcM`) on the main street's arc path using the functions from
   `shared/cleaning-geometry.ts`.
3. Writes `data/cleaning-segments.json` with resolved segments.
4. Writes `data/cleaning-segments.audit.txt` listing every problem row with reason and
   suggested action.

The runtime renderer (CF-22) will use `cleaning-segments.json` directly. Do not change
any runtime rendering code in this feature.

## Output data shape

### `data/cleaning-segments.json`
```typescript
type CleaningSegmentsFile = {
  roadGeometryHash: string;  // SHA-1 of road-geometry.json content (build-time only)
  segments: CleaningSegment[];
};

type CleaningSegment = {
  id: string;              // stable slug: "hudson-st-east-14th-st-to-15th-st"
  streetKey: string;       // normalized: "HUDSON ST"
  side: string;            // "East" | "West" | "North" | "South" | "Both"
  fromLabel: string;       // normalized cross-street key: "14TH ST"
  toLabel: string;         // normalized cross-street key: "15TH ST"
  fromArcM: number;        // arc meters from street start (absent if geometry override)
  toArcM: number;          // arc meters from street start (absent if geometry override)
  geometry?: [number, number][];  // explicit lat/lng fallback (manual override only)
  schedule: string;        // original schedule string, unchanged
  confidence: "auto" | "manual";
};
```

Segments with both `fromArcM`/`toArcM` and `geometry` use arc range (arc takes precedence).
Segments with only `geometry` (no arc range) are explicit-geometry overrides.

Add these types to `shared/types.ts` so CF-22 can import them.

### `data/cleaning-segment-overrides.json`
Initial content is an empty array `[]`. This file is committed to source so the Creator
agent does not need to populate it — it starts empty and a human fills in hard cases after
reviewing the audit report. Format:

```typescript
type CleaningSegmentOverride = {
  match: {
    streetKey: string;
    side: string;
    fromLabel: string;
    toLabel: string;
  };
  fromArcM?: number;
  toArcM?: number;
  geometry?: [number, number][];
  confidence: "manual";
  note: string;
};
```

## Build script algorithm (`scripts/build-cleaning-segments.ts`)

Run via `npx tsx scripts/build-cleaning-segments.ts` from the project root.

For each entry in `data/street-cleaning.json`:

**Step 1 — Normalize**
- `streetKey = normalizeToGeometryKey(entry.street)` (from `shared/cleaning-geometry.ts`)
- Parse location string with regex `/^(.+?)\s+to\s+(.+?)$/i` (covers "X to Y" and "X and Y")
- `fromLabel = normalizeToGeometryKey(rawFrom)`
- `toLabel = normalizeToGeometryKey(rawTo)`
- Generate stable `id` from `${streetKey}-${side}-${fromLabel}-${toLabel}` lowercased with
  spaces replaced by `-`.

**Step 2 — Check override**
Look for an entry in `data/cleaning-segment-overrides.json` where `match.streetKey`,
`match.side`, `match.fromLabel`, `match.toLabel` all match. If found, use override and
set `confidence: "manual"`. Skip auto-resolution.

**Step 3 — Auto-resolve arc positions**
1. Get `rawWays = roadGeometry[streetKey]`. If missing → log "MISSING_ROAD_GEOMETRY" and continue.
2. `ways = filterWaysByStreetLat(rawWays, streetKey)` (from `shared/cleaning-geometry.ts`).
   If empty after filter → log "FILTERED_TO_EMPTY".
3. `{ points, cumArc, wayEnds } = flattenWaysToArcPath(ways)`.
4. For each of `fromLabel` and `toLabel`:
   - Get `crossWays = roadGeometry[crossLabel]`. If missing → log "MISSING_CROSS_STREET".
   - **Project each cross-street way point onto the main street's arc path** using
     closest-point-on-segment math:
     - For each arc segment `[points[i], points[i+1]]`, compute the closest point on that
       segment to the cross-street way point.
     - Track the minimum distance and the corresponding cumulative arc distance at the
       projected point.
     - After iterating all cross-street way points and all arc segments, take the overall
       minimum-distance projection.
   - If minimum distance > 30m → log "SUSPECT_SNAP" (keep result but flag it).
   - Record `arcM` at the projection as `fromArcM` or `toArcM`.
5. Ensure `fromArcM < toArcM` (swap if reversed).
6. Check for disconnected way gap: if any `wayEnds` index falls strictly between `fromArcM`
   and `toArcM`, log "SPANS_WAY_GAP".
7. If `toArcM - fromArcM < 20` → log "TOO_SHORT".
8. If `toArcM - fromArcM > 1500` → log "TOO_LONG".
9. If `fromArcM < 10` and `toArcM > cumArc[cumArc.length - 1] - 10` → log "WHOLE_STREET".

**Step 4 — Collapse duplicate schedules**
Multiple cleaning rows with the same `{streetKey, side, fromLabel, toLabel}` produce one
`CleaningSegment` per unique `{id}`. Collect all `schedule` strings into an array — but
the segment's `schedule` field stores only the first encountered. The runtime picks the
highest-priority schedule among multiple segments with the same `id` at render time.
For the purpose of this builder, just emit one segment per unique id with its schedule;
if duplicates exist, emit both (they have the same id and geometry — the runtime dedupes
by highest-priority status).

**Step 5 — Write outputs**
Compute `roadGeometryHash` = SHA-1 of `data/road-geometry.json` file content using Node's
`crypto.createHash('sha1').update(fileContent).digest('hex')`.

Write `data/cleaning-segments.json`:
```json
{
  "roadGeometryHash": "<sha1>",
  "segments": [...]
}
```

Write `data/cleaning-segments.audit.txt` with sections:
```
=== RESOLVED AUTO (n) ===
<id>  arcM: 123.4 → 456.7

=== RESOLVED MANUAL OVERRIDE (n) ===
<id>  [arc-range | explicit-geometry]  note: ...

=== MISSING_ROAD_GEOMETRY (n) ===
streetKey | side | schedule | location | reason

=== MISSING_CROSS_STREET (n) ===
streetKey | side | label | location | reason

=== SUSPECT_SNAP (n) ===
id | label | distM | arcM

=== SPANS_WAY_GAP (n) ===
id | fromArcM | toArcM | gapAt

=== TOO_SHORT (n) ===
id | length

=== TOO_LONG (n) ===
id | length

=== WHOLE_STREET (n) ===
id | fromArcM | toArcM
```

## Cross-street projection helper

The hardest part is `projectCrossStreetToArc`. This should be a standalone named function
in the build script (not in shared, since it's builder-only logic):

```typescript
function projectCrossStreetToArc(
  crossWays: [number, number][][],
  arcPoints: [number, number][],
  arcCumDist: number[]
): { arcM: number; distM: number } | null
```

For correctness: approximate lat/lng degrees to meters using `latDegToM = 111320` and
`lngDegToM = 111320 * Math.cos(refLat * Math.PI / 180)` where `refLat` is the centroid
latitude of the arc path. Use these scale factors to convert degree differences to meters
for the distance calculations. Do not use Haversine — the approximation is sufficient at
Hoboken's scale (<3km) and keeps the function simple.

Closest-point-on-segment algorithm for a point `P` and segment `[A, B]`:
- `t = clamp(dot(P-A, B-A) / dot(B-A, B-A), 0, 1)`
- `proj = A + t * (B - A)` (in meter space)
- `distM = |P - proj|` (Euclidean in meter space)
- `arcM = cumDist[i] + t * (cumDist[i+1] - cumDist[i])` where `i` is the arc segment index

## Behavioral contracts

- GIVEN a cleaning row with location "Fourteenth St. to Fifteenth St." on "HUDSON ST",
  WHEN the builder resolves it using synthetic road-geometry data with known Hudson St ways
  and 14TH ST / 15TH ST cross-street ways placed one block apart (~130m),
  THEN `toArcM - fromArcM` is between 100 and 200 (one city block).

- GIVEN a cross-street way whose closest point to the main street arc is 40m away,
  WHEN the builder processes the row,
  THEN the result has `confidence: "auto"` and the audit includes a SUSPECT_SNAP entry for it.

- GIVEN a main street arc path with a gap of 60m between way 1 and way 2, and `fromArcM`
  falls in way 1 and `toArcM` falls in way 2,
  WHEN the builder processes the row,
  THEN the audit includes a SPANS_WAY_GAP entry for the segment.

- GIVEN a cleaning row matching an entry in `cleaning-segment-overrides.json` with
  `fromArcM: 320, toArcM: 890`,
  WHEN the builder processes the row,
  THEN the output segment has `fromArcM: 320, toArcM: 890, confidence: "manual"` regardless
  of what auto-resolution would produce.

- GIVEN a cleaning row matching an entry in `cleaning-segment-overrides.json` with explicit
  `geometry` (no fromArcM/toArcM),
  WHEN the builder processes the row,
  THEN the output segment has `geometry: [...]` and no `fromArcM`/`toArcM` field,
  AND `confidence: "manual"`.

- GIVEN two cleaning rows with the same street/side/location but different schedule strings,
  WHEN the builder processes them,
  THEN the output contains two segments with the same `id` and different `schedule` strings.

- GIVEN `data/road-geometry.json` exists,
  WHEN the builder runs,
  THEN the output JSON contains a non-empty `roadGeometryHash` string.

- GIVEN all auto-resolved segments,
  THEN every segment has `fromArcM < toArcM` (never equal, never reversed).

## Test list

Tests use synthetic geometry — do not read actual data files in tests.

1. `projectCrossStreetToArc` — perpendicular cross-street projected correctly onto arc
2. `projectCrossStreetToArc` — single-point cross-street (sparse) still projects
3. `projectCrossStreetToArc` — cross-street 40m away → returns result with distM > 30
4. Segment resolves correctly: fromArcM < toArcM, block length 100–200m (synthetic one-block setup)
5. Gap >50m in arc path flagged as SPANS_WAY_GAP in audit
6. Segment with fromArcM-toArcM < 20m flagged as TOO_SHORT
7. Arc-range override replaces auto result: output uses override fromArcM/toArcM
8. Explicit-geometry override: output has geometry field, no fromArcM/toArcM
9. Duplicate schedule rows → two segments with same id, different schedules, both emitted
10. Output includes non-empty roadGeometryHash
11. fromArcM < toArcM for all auto-resolved segments in synthetic data
