# CF-16 — Address Arc Index

## What this is

Adds precise address-range highlighting capability by persisting OSM address-node data as arc positions at build time. This enables any caller to draw a street highlight for a specific house-number range (e.g. "729–740 Bloomfield St") clipped accurately to road geometry.

**Key design principle:** OSM address nodes alternate odd/even sides (zig-zag in lat/lng space). Projecting them to the road centerline at build time and storing the arc position (meters from road start) collapses this to a stable 1-D coordinate. The rendered highlight still uses road geometry — not address-node positions.

## `fetcher/build-street-parity.ts`

### Changes

1. **Refactor `computeDotForPoint` into `snapAddressNodeToRoad`.**

   The existing function computes the nearest-segment projection internally but discards everything except the dot product. Replace it with:

   ```typescript
   function snapAddressNodeToRoad(
     ways: [number, number][][],
     lat: number,
     lng: number
   ): { dot: number; distanceM: number; arcM: number } | null
   ```

   Implementation:
   - Call `flattenWaysToArcPath(ways)` (defined below) to get `{ points, cumArc }`.
   - Project `(lat, lng)` onto each segment of the flat path using the same perpendicular-projection math as the old function.
   - Track: `bestDist`, `bestDot`, `bestArcM` (cumulative arc at the projection point).
   - Return `null` if `bestDist > 50` (same threshold as before).
   - Return `{ dot: bestDot, distanceM: bestDist, arcM: bestArcM }`.

   Parity voting continues unchanged: use `result.dot`.

2. **Add `flattenWaysToArcPath` helper.**

   ```typescript
   function flattenWaysToArcPath(
     ways: [number, number][][]
   ): { points: [number, number][]; cumArc: number[]; wayEnds: number[] }
   ```

   - Concatenates all ways in order into one flat `points` array.
   - `cumArc[i]` = cumulative arc-length in metres from `points[0]` to `points[i]`.
   - `wayEnds[w]` = index of the last point of way `w` in the flat array (used to preserve way-boundary information downstream).
   - Uses flat-earth distance: `111320 m/degree` lat, `111320 * cos(lat) m/degree` lng.

   > **Critical:** The identical algorithm must be duplicated into `app/map.ts`. Both copies must produce the same arcM for the same input — this is the contract that makes build-time arcM values usable at runtime.

3. **Accumulate address-arc entries during the node loop.**

   While iterating address nodes (the existing loop used for parity voting), also:
   - Call `snapAddressNodeToRoad` (which now returns `arcM`).
   - Parse `houseNum` from `addr:housenumber` (same regex as before: `/^(\d+)/`).
   - Accumulate `[houseNum, arcM]` per streetKey in a `Map<string, [number, number][]>`.

4. **Collapse duplicate house numbers by median.**

   After the loop, for each street:
   - Group entries by houseNum.
   - If multiple entries share the same houseNum, take the **median arcM** (sort the arcM values, take the middle element; for even-length arrays, take the lower middle).
   - This handles apartments, building entrances, and repeated OSM nodes.

5. **Sort and write.**

   - Sort each street's `[houseNum, arcM][]` by houseNum ascending.
   - Write `data/address-arc.json` using `JSON.stringify(result, null, 2)`.

   Add a comment above the write:
   ```typescript
   // address-arc.json is computed against the current road-geometry.json.
   // If road-geometry.json is regenerated, re-run build-street-parity to keep arcM values valid.
   ```

6. **`runBuildStreetParity` now writes two files:** `data/street-parity.json` and `data/address-arc.json`. Log both paths to console on success.

### Exports

The existing `normalizeStreet` export is unchanged. No new exports are required from this file.

## `app/map.ts`

Add five new items. Do not modify any existing function.

### 1. `flattenWaysToArcPath` (private)

```typescript
function flattenWaysToArcPath(
  ways: [number, number][][]
): { points: [number, number][]; cumArc: number[]; wayEnds: number[] }
```

Identical algorithm to the one in `build-street-parity.ts`. This is the shared arc coordinate system — build and runtime must produce the same output for the same input.

### 2. `initAddressArcIndex`

```typescript
export function initAddressArcIndex(data: AddressArcIndex): void
```

Stores `data` in a module-level `let _addressArcIndex: AddressArcIndex = {}`. Same pattern as `initRoadGeometry` / `initStreetParity`.

### 3. `interpolateHouseNumToArcM`

```typescript
export function interpolateHouseNumToArcM(
  streetKey: string,
  houseNum: number
): { arcM: number; clamped: boolean } | null
```

- Look up `_addressArcIndex[streetKey]`. Return `null` if absent or empty.
- Binary search (or linear scan) for the two entries that bracket `houseNum`.
- If `houseNum` is below the minimum known houseNum: return `{ arcM: entries[0][1], clamped: true }`.
- If `houseNum` is above the maximum: return `{ arcM: entries[last][1], clamped: true }`.
- Otherwise linearly interpolate: `arcM = lo.arcM + (houseNum - lo.num) / (hi.num - lo.num) * (hi.arcM - lo.arcM)`, return `{ arcM, clamped: false }`.

### 4. `clipWaysToArcRange`

```typescript
export function clipWaysToArcRange(
  ways: [number, number][][],
  fromArcM: number,
  toArcM: number
): [number, number][][]
```

- Call `flattenWaysToArcPath(ways)` to get `{ points, cumArc, wayEnds }`.
- Walk `points` in order, accumulating arc. When entering `[fromArcM, toArcM]`, interpolate the entry point. Collect interior points. When exiting, interpolate the exit point.
- **Preserve way boundaries:** whenever the walk index crosses a `wayEnds` boundary, close the current output segment and start a new one — never emit a segment that spans a gap between disconnected ways.
- Only emit a segment with ≥2 points.
- Return the array of clipped way segments.

### 5. `renderAddressRangeHighlight`

```typescript
export function renderAddressRangeHighlight(
  street: string,
  fromNum: number,
  toNum: number,
  color: string,
  opacity: number,
  opts?: {
    side?: "East" | "West" | "North" | "South";
    fallbackLat?: number;
    fallbackLng?: number;
  }
): LeafletPolyline[]
```

Orchestrator:

1. `streetKey = normalizeToGeometryKey(street)`
2. `allWays = _roadGeometry[streetKey]` — return `[]` if absent.
3. `from = interpolateHouseNumToArcM(streetKey, fromNum)`
4. `to   = interpolateHouseNumToArcM(streetKey, toNum)`
5. If both are non-null:
   - `lo = Math.min(from.arcM, to.arcM)` (direction not guaranteed)
   - `hi = Math.max(from.arcM, to.arcM)`
   - `clipped = clipWaysToArcRange(allWays, lo, hi)`
   - `merged  = mergeWays(clipped)`
6. Else if `opts?.fallbackLat !== undefined && opts?.fallbackLng !== undefined`:
   - `addrRange = Math.max(0, toNum - fromNum)`
   - `halfLengthM = Math.max(5, (addrRange / 2 + 1) * 4)`
   - `merged = [getSubsegment(allWays, opts.fallbackLat, opts.fallbackLng, halfLengthM)]`
7. Else return `[]`.
8. Determine `forcedDir`:
   - Primary: derive from `opts.side` (e.g. `"East"` on an N-S street → `dir = 1`).
   - Secondary: if `opts.fallbackLat/Lng` provided, dot-product against the road.
   - Fallback: `_streetParity[streetKey]` + parity of `fromNum` (unreliable for mixed ranges — use only as last resort).
9. For each segment in `merged`, call `offsetPolylinePoints(pts, opts?.fallbackLat ?? pts[0][0], opts?.fallbackLng ?? pts[0][1], LATERAL_OFFSET_M, forcedDir)` and create `L.polyline(offset, { color, weight: 4, opacity })`.
10. Return the array of polylines.

## Test list (`tests/unit/address-arc.test.ts`)

Tests run in `environment: "node"`. Use synthetic data — no real `address-arc.json` or road geometry required.

**`interpolateHouseNumToArcM` — exact interpolation**
Given a streetKey with entries `[[100, 0.0], [200, 50.0]]`,
When `interpolateHouseNumToArcM(key, 150)` is called,
Then it returns `{ arcM: 25.0, clamped: false }`.

**`interpolateHouseNumToArcM` — clamp low**
Given entries `[[100, 0.0], [200, 50.0]]`,
When called with houseNum `50`,
Then it returns `{ arcM: 0.0, clamped: true }`.

**`interpolateHouseNumToArcM` — clamp high**
Given entries `[[100, 0.0], [200, 50.0]]`,
When called with houseNum `300`,
Then it returns `{ arcM: 50.0, clamped: true }`.

**`interpolateHouseNumToArcM` — unknown street**
Given no entry for streetKey `"UNKNOWN ST"`,
When called,
Then it returns `null`.

**`interpolateHouseNumToArcM` — exact match**
Given entries `[[100, 10.0], [200, 60.0]]`,
When called with houseNum `100`,
Then it returns `{ arcM: 10.0, clamped: false }`.

**`clipWaysToArcRange` — basic clip**
Given a single way `[[0,0],[0,0.001],[0,0.002]]` (roughly 100m+ long),
When clipped to `[30, 70]` (metres),
Then the result has one segment with ≥2 points whose first point is ~30m from start and last is ~70m.

**`clipWaysToArcRange` — does not bridge way gap**
Given two disconnected ways: way A `[[40.740,-74.030],[40.741,-74.030]]` and way B `[[40.745,-74.030],[40.746,-74.030]]`,
When clipped to an arc range that spans across the gap between the two ways,
Then the result contains two separate segments (one per way), not one continuous segment.

**`clipWaysToArcRange` — entirely outside range**
Given a single way and a clip range beyond its total length,
Then the result is an empty array.

**`clipWaysToArcRange` — min/max swap**
Given `fromArcM = 70, toArcM = 30` (reversed),
Then the result is identical to calling with `fromArcM = 30, toArcM = 70` (Math.min/max applied internally).

**`flattenWaysToArcPath` — wayEnds tracking**
Given two ways of 3 points each,
When `flattenWaysToArcPath` is called,
Then `wayEnds[0] === 2` (last index of way 0) and `wayEnds[1] === 5` (last index of way 1).

## Hard constraints

- No `as any`. No `!`. Named exports only. No default exports.
- `flattenWaysToArcPath` in `build-street-parity.ts` and `app/map.ts` must use identical arithmetic — same cos-lat formula, same segment-length accumulation. The contract is that arcM for a given node is the same value whether computed at build time or at runtime.
- `clipWaysToArcRange` must never emit a point array that spans two disconnected ways.
- `renderAddressRangeHighlight` must use `LeafletPolyline[]` as the return type (the local shim type defined in `app/map.ts`), not `L.Polyline[]`.
- Tests run in `environment: "node"`. No DOM, no Leaflet imports in the test file.
- Do not run `npm run build` during implementation. The Verifier runs it separately.
