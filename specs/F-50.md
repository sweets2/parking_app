# F-50 — Required Parking Data Loader

## Purpose

Ensure parking data required by Check and Rules is loaded before createApp() enters ready state.

## Mode affected

Both.

## State fields read

- allSigns
- activeSigns
- parkingSegments

## State fields written

- allSigns
- activeSigns
- parkingSegments

## TypeScript contract signatures

No new module required unless it simplifies the implementation.

If creating a helper, use:

```typescript
export interface ParkingRuntimeData {
  signs: Sign[];
  activeSigns: Sign[];
  cleaningEntries: StreetCleaningEntry[];
  garages: Garage[];
  snowRoutes: SnowRoute[];
  roadGeometry?: RoadGeometry;
  parkingSegments: ParkingSegment[];
}

export async function loadParkingRuntimeData(): Promise<ParkingRuntimeData>;
```

`createApp` must receive `parkingSegments` as part of its `initialData` argument.
Update the `initialData` parameter type in `createApp` to include:

```typescript
parkingSegments: ParkingSegment[];
```

so that `AppState` in ready mode carries `parkingSegments: ParkingSegment[]`.

## state/event/rendering behavior

Before createApp() reaches ready mode:
- signs must be loaded or safely defaulted
- activeSigns must be derived from the loaded signs using filterLoadTimeNoise + filterActive
- cleaning entries must be loaded or safely defaulted (empty array on fetch failure)
- snow routes must be loaded or safely defaulted (empty array on fetch failure)
- road geometry should be loaded if available (undefined on fetch failure)
- parkingSegments must be built using buildParkingSegmentCatalog({signs, cleaningEntries, snowRoutes, roadGeometry}) before createApp() is called

Do not fire-and-forget required data that Check/Rules needs (signs, parkingSegments).

Allowed:
- cross-streets.json may remain fire-and-forget (geocode cache, enrichment only)
- garages.json may remain fire-and-forget (display only)
- street-parity.json must be awaited (already is, per CLAUDE.md F-41/F-42 gotcha)
- road-geometry.json must be awaited (already is)

## Required vs optional fetches

The following fetches MUST complete (await or catch-with-default) before createApp() is called:
- `data/latest.json` — required; if unavailable, show error banner and return early (no createApp call)
- `data/street-cleaning.json` — required for parkingSegments; catch failure and default to `[]`
- `data/snow-emergency-routes.json` — required for parkingSegments; catch failure and default to `[]`
- `data/road-geometry.json` — required for segment geometry; catch failure and default to `undefined`
- `data/street-parity.json` — required for sign positions; catch failure and ignore

The following fetches MAY remain fire-and-forget (after createApp() is called):
- `data/cross-streets.json` — geocode cache seed, non-fatal
- `data/garages.json` — display only markers, non-fatal
- `data/future.json` — upcoming signs, non-fatal

## Acceptance criteria

Given required static data files (latest.json, street-cleaning.json, snow-emergency-routes.json) are available
When the app starts
Then createApp receives an initialData object whose parkingSegments field has length greater than 0 (specifically, length equal to the number of unique (street, side, location) tuples produced by buildParkingSegmentCatalog from the loaded signs, cleaningEntries, and snowRoutes).

Given a latest.json fetch fails
When the app starts
Then createApp is NOT called and the error banner element's textContent is set to a non-empty string (e.g. "Failed to load parking data.").

Given signs load successfully from latest.json with N signs
When ready state is created
Then createApp receives an initialData object where initialData.signs has length N (all raw signs before filterLoadTimeNoise) and initialData.activeSigns (if present in initialData) or the resulting AppState.activeSigns equals filterActive(filterLoadTimeNoise(signs, fetchTime), fetchTime).

## Required fix for tests/unit/f27.test.ts (regression from main.ts change)

F-50 adds a call to `buildParkingSegmentCatalog` inside `initBrowserApp` after `correctSignPositions` is called. This causes a regression in `tests/unit/f27.test.ts` because that test file's `beforeEach` calls `vi.resetAllMocks()` which clears the inline `vi.fn()` implementations in the `vi.mock("../../app/map", ...)` factory — specifically `correctSignPositions: vi.fn((signs: unknown[]) => signs)` becomes `vi.fn()` returning `undefined`. After `correctSignPositions` returns `undefined`, `signsData.signs` becomes `undefined`, and `buildParkingSegmentCatalog` receives `undefined` for signs, causing "TypeError: signs is not iterable".

Fix required in `tests/unit/f27.test.ts`:

1. Extract `correctSignPositions` and `buildParkingSegmentCatalog` as module-level `vi.fn()` variables (with their initial implementations):

```typescript
const mockCorrectSignPositionsF27 = vi.fn((signs: unknown[]) => signs);
const mockBuildParkingSegmentCatalogF27 = vi.fn((): import("../../shared/types").ParkingSegment[] => []);
```

2. Reference them in their respective `vi.mock` factories instead of inline `vi.fn(...)` calls:

In `vi.mock("../../app/map", ...)`, replace:
```typescript
correctSignPositions: vi.fn((signs: unknown[]) => signs),
```
with:
```typescript
correctSignPositions: mockCorrectSignPositionsF27,
```

In `vi.mock("../../shared/segment-catalog", ...)`, replace:
```typescript
buildParkingSegmentCatalog: vi.fn(() => []),
```
with:
```typescript
buildParkingSegmentCatalog: mockBuildParkingSegmentCatalogF27,
```

3. In the `beforeEach` of `describe("F-27.3 locate button wiring in initBrowserApp", ...)`, restore both implementations after `vi.resetAllMocks()`:

```typescript
mockCorrectSignPositionsF27.mockImplementation((signs: unknown[]) => signs);
mockBuildParkingSegmentCatalogF27.mockImplementation((): import("../../shared/types").ParkingSegment[] => []);
```

This ensures that after `vi.resetAllMocks()` clears the implementations, they are restored before each F-27.3 test runs `initBrowserApp`.
