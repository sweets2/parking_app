# F-46B — Remove Dev Time Override / Production Safety Cleanup

## Purpose

Remove development-only time/location hacks before Check | Rules features depend on parking evaluation.

## Mode affected

Both.

## State fields read

Existing app state only.

## State fields written

No new persistent state.

## TypeScript contract signatures

No new exported contracts required. The following internal symbols are removed from `app/main.ts`:

```typescript
// REMOVE these from app/main.ts:
// const DEV_FORCE_NOW: Date | null = ...
// const DEV_TEST_STREET = "Washington St."
// function devNow(): Date { ... }
// function devEntries(entries: StreetCleaningEntry[]): StreetCleaningEntry[] { ... }
```

After removal every call site that previously called `devNow()` must call `new Date()` directly, and every call site that previously called `devEntries(cleaningEntries)` must pass `cleaningEntries` directly.

## state/event/rendering behavior

Delete the dev-override block from `app/main.ts` entirely (lines containing `DEV_FORCE_NOW`, `DEV_TEST_STREET`, `devNow`, `devEntries`). Replace every call site:

- `devNow()` → `new Date()`
- `devEntries(cleaningEntries)` → `cleaningEntries`

This affects the following call sites in `app/main.ts`:
- `renderState` inner function — `const now = devNow()` → `const now = new Date()`
- `renderState` inner function — `renderViolationHighlights(devEntries(cleaningEntries), now)` → `renderViolationHighlights(cleaningEntries, now)`
- `scheduleViolationRefresh` timeout callback — `renderViolationHighlights(devEntries(cleaningEntries), devNow())` → `renderViolationHighlights(cleaningEntries, new Date())`
- `initBrowserApp` / `silentRefresh` — any remaining `devNow()` or `devEntries(...)` usages

Allowed:
- Tests may inject stable fixture dates via function parameters (e.g. passing `NOW_STABLE` to pure-logic functions).
- Pure logic modules may accept `Date` parameters.
- UI may display current time from `new Date()` at event-handling time.

## Acceptance criteria

Given the app starts normally and `cleaningEntries` contains entries for multiple streets (e.g. "Washington St." and "Observer Hwy.")
When `renderState` is called (parking data loads)
Then `renderViolationHighlights` is called with the full `cleaningEntries` array (length ≥ 2, containing entries for streets other than "Washington St."), not a Washington-St-filtered subset — i.e. `devEntries` no longer exists and `renderViolationHighlights` receives `cleaningEntries` directly.

Given `cleaningEntries` contains entries for streets other than "Washington St."
When `renderState` fires after parking data loads
Then `renderViolationHighlights` receives an array argument that includes at least one entry whose `street` field is NOT "Washington St." — confirming the Washington-St-only filter is removed.

Given tests need deterministic time for `showStreetPopup` and `renderViolationHighlights`
When a test calls `initBrowserApp` and then manually invokes the captured `renderState` callback or `scheduleViolationRefresh` timeout
Then no reference to `DEV_FORCE_NOW`, `devNow`, or `devEntries` exists in `app/main.ts` — the test uses `vi.useFakeTimers()` or passes `NOW_STABLE` directly to the pure-logic functions under test.

Given `scheduleViolationRefresh` fires its hourly timeout callback
When `getState()` returns a non-loading, non-error state
Then `renderViolationHighlights` is called with `cleaningEntries` (unfiltered) and `new Date()` (real runtime clock, not a dev-forced value).
