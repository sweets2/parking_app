# F-48 — Exact Window Conflict Evaluator + Next Restriction

## Purpose

Evaluate whether a parking segment conflicts with a requested parking window and identify the next relevant restriction without fixed time sampling.

## Mode affected

Check and Rules. Pure domain logic.

## State fields read

None.

## State fields written

None.

## Import rule for parking-logic.ts

`shared/parking-logic.ts` may only import from `../shared/types` (the one allowed import source). All types used by the three exported functions (`ParkingSegment`, `CheckQuery`, `CheckResultSegment`, `ParkingWindowConflict`, `NextRestriction`) are defined in `shared/types.ts` and must be imported as named type-only imports:

```typescript
import type { Sign, ParkingSegment, CheckQuery, CheckResultSegment, ParkingWindowConflict, NextRestriction } from "../shared/types";
```

No other import sources are permitted in `shared/parking-logic.ts`.

## TypeScript contract signatures

Update shared/parking-logic.ts.

Required exports:

```typescript
export function evaluateParkingWindow(
  segment: ParkingSegment,
  query: CheckQuery
): CheckResultSegment;

export function getPrimaryConflict(
  conflicts: ParkingWindowConflict[]
): ParkingWindowConflict | undefined;

export function getNextRestriction(
  segment: ParkingSegment,
  after: Date
): NextRestriction | undefined;
```

## Behavior

Do not use 30-minute sampling.

A restriction conflicts if any expanded interval overlaps:
[query.startTime, query.endTime)

Interval policy:
- start inclusive
- end exclusive

Priority order:
1. tow
2. snow
3. ticket
4. limited
5. unknown
6. safe

If multiple conflicts overlap, primaryConflict is the highest-priority conflict.

If no conflicts exist:
- status is "safe"
- conflicts is []
- primaryConflict is undefined

If data is insufficient:
- status may be "unknown"
- conflicts contains one entry with status "unknown" and a reason string explaining why.

`getNextRestriction` semantics:
- `after` is exclusive: the returned restriction must have `startsAt > after`.
- When multiple future restrictions exist, return the one with the smallest `startsAt`.
- Returns `undefined` when no future restriction exists.

## Helper: restriction interval expansion

Street-cleaning entries are parsed into time intervals using `parseScheduleRange` (already in parking-logic.ts). Each active cleaning day in the given week that starts after `after` produces a candidate restriction. Tow signs produce a restriction interval from `sign.start_iso` to `sign.end_iso`. Snow routes produce no time-bound restriction (they are seasonal and always-active; `getNextRestriction` may ignore them or return undefined for a segment with only snow routes).

## Acceptance criteria

Given a ParkingSegment with a street-cleaning interval 10:00–12:00 on a specific day
When query.startTime is that day at 09:00 and query.endTime is that day at 10:00
Then `result.status === "safe"` and `result.conflicts.length === 0`.

Given a ParkingSegment with a street-cleaning interval 10:00–12:00 on a specific day
When query.startTime is that day at 09:59 and query.endTime is that day at 10:01
Then `result.status === "ticket"` and `result.conflicts.length === 1` and `result.conflicts[0].status === "ticket"`.

Given a ParkingSegment with a street-cleaning interval 10:00–12:00 on a specific day
When query.startTime is that day at 12:00 and query.endTime is that day at 13:00
Then `result.status === "safe"` and `result.conflicts.length === 0` (end is exclusive: [10:00,12:00) does not overlap [12:00,13:00)).

Given a ParkingSegment that has both a tow sign active during the query and a street-cleaning interval that overlaps the query
When evaluateParkingWindow is called
Then `result.status === "tow"` and `result.primaryConflict !== undefined` and `result.primaryConflict.status === "tow"`.

Given a ParkingSegment that has a snow route and a cleaning interval both overlapping the query (no tow sign)
When evaluateParkingWindow is called
Then `result.status === "snow"` and `result.primaryConflict !== undefined` and `result.primaryConflict.status === "snow"`.

Given a ParkingSegment with no cleaning entries, no tow signs, and no snow routes
When evaluateParkingWindow is called for any query window
Then `result.status === "safe"` and `result.conflicts.length === 0` and `result.primaryConflict === undefined`.

Given a ParkingSegment with cleaningEntries that have an unparseable schedule string
When evaluateParkingWindow is called
Then `result.status === "unknown"` and `result.conflicts.length === 1` and `result.conflicts[0].status === "unknown"`.

Given getPrimaryConflict is called with an empty array
When evaluated
Then the return value is `undefined`.

Given getPrimaryConflict is called with conflicts of status ["ticket", "tow", "limited"]
When evaluated
Then the return value is the conflict whose `status === "tow"` (highest priority).

Given getPrimaryConflict is called with conflicts of status ["snow", "limited"]
When evaluated
Then the return value is the conflict whose `status === "snow"`.

Given a ParkingSegment with a tow sign whose start_iso is in the future relative to `NOW_STABLE`
When getNextRestriction is called with `after = NOW_STABLE`
Then the return value is not undefined and `result.startsAt` equals the tow sign's start Date and `result.status === "tow"`.

Given a ParkingSegment with a tow sign whose start_iso equals `NOW_STABLE` (not strictly after)
When getNextRestriction is called with `after = NOW_STABLE`
Then the return value is `undefined` (after is exclusive — startsAt must be strictly greater than after).

Given a ParkingSegment with no tow signs, no cleaning entries, and no snow routes
When getNextRestriction is called
Then the return value is `undefined`.

## Time-sensitive test discipline

**Critical:** Do NOT import from `tests/fixtures/signs.ts` in the F-48 test file. That module executes `readFileSync("data/latest.json")` at module scope, which crashes the entire test suite when `data/latest.json` is absent (CI / fresh checkout).

All new F-48 tests MUST go into a separate file: `tests/unit/parking-logic-window.test.ts`.

Define a local constant at the top of that file instead of importing from the fixture:

```typescript
// Fixed reference point — 2026-06-09 noon ET (UTC-4) = 2026-06-09T16:00:00.000Z
const NOW_STABLE = new Date("2026-06-09T16:00:00.000Z");
```

Use `NOW_STABLE` (the local constant) as the `after` argument for `getNextRestriction` tests. Tow sign start/end ISO strings in test fixtures must use explicit UTC Z-suffix strings derived from `NOW_STABLE.getTime()` offsets.

Do NOT add any import from `tests/fixtures/signs.ts` to `tests/unit/parking-logic-window.test.ts`.
